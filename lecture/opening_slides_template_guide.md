# Opening Slides HTML Template Guide

이 문서는 [`opening_slides_template.html`](./opening_slides_template.html)을 강의용 동적 HTML 프레젠테이션 템플릿으로 재사용하기 위한 작업 가이드다. 기존 HTML 파일은 템플릿 원본으로 보존하고, 새 강의 오프닝을 만들 때는 복사본을 만들어 내용과 필요한 레이아웃만 교체한다.

## 목적

- 강의 첫 3-7분에 사용할 오프닝 프레젠테이션을 빠르게 만든다.
- 키보드, 터치, 전체화면, 진행바가 있는 단일 HTML 산출물을 만든다.
- PDF 출력이 가능한 16:9 슬라이드 구조를 유지한다.
- 주제만 바뀌어도 같은 시각 품질과 발표 흐름을 재사용한다.

## 원본 구조

`opening_slides_template.html`은 세 부분으로 구성된다.

| 영역 | 위치 | 역할 |
|---|---:|---|
| `<head>` | 상단 | 문서 메타, Pretendard 웹폰트, 전체 CSS |
| `.stage > .deck` | 본문 | 1280x720 고정 슬라이드 캔버스 |
| `<script>` | 하단 | 슬라이드 이동, 진행률, 점 네비게이션, 전체화면, 터치 스와이프 |

핵심 DOM 구조는 유지한다.

```html
<div class="stage">
  <div class="deck" id="deck">
    <section class="slide cover active">...</section>
    <section class="slide">...</section>
  </div>
</div>
```

`section.slide` 개수는 JS가 자동으로 읽는다. 슬라이드를 추가하거나 삭제해도 진행바, 점 네비게이션, 페이지 카운트는 자동 갱신된다.

## 고정 규칙

템플릿 재사용 시 아래 항목은 기본적으로 유지한다.

| 항목 | 규칙 |
|---|---|
| 캔버스 | `.deck`은 `1280px x 720px` 고정 |
| 스케일링 | `fit()` 함수가 뷰포트에 맞춰 `scale()` 적용 |
| 슬라이드 | 모든 장은 `section.slide`로 작성 |
| 첫 장 | 첫 슬라이드 하나만 `active` 클래스를 가진다 |
| 애니메이션 | 등장 요소에는 `data-anim`과 `--d:n` 지연값 사용 |
| 진행 UI | `.progress`, `.chrome`, `.dots`, `.ctl`, `.count` 유지 |
| 출력 | `@media print` 블록 유지 |
| 폰트 | Pretendard Variable CDN과 시스템 fallback 유지 |

## 디자인 토큰

CSS `:root`의 변수는 템플릿의 시각 언어다.

| 토큰 | 값 | 용도 |
|---|---|---|
| `--bg` | `#F5F7FC` | 전체 배경 |
| `--ink` | `#0F1A2C` | 제목, 핵심 문장 |
| `--body` | `#37435C` | 본문 |
| `--muted` | `#5E6B85` | 보조 설명 |
| `--faint` | `#98A3B7` | 라벨, 힌트 |
| `--line` | `rgba(16,26,44,.10)` | 카드/패널 경계선 |
| `--card` | `#FFFFFF` | 카드 배경 |
| `--azure` | `#2E6BFF` | 코드, 기준, 첫 축 |
| `--violet` | `#7C4DFF` | AI, 해석, 두 번째 축 |
| `--amber` | `#F0A62A` | 사람, 결정, 세 번째 축 |
| `--rose` | `#F0616B` | 문제, 충돌, 경고 |

색을 새로 추가하기보다 위 토큰의 의미를 유지해서 주제에 맞게 매핑한다. 예를 들어 새 강의가 세무 자동화라면 `--azure`는 "계산/규칙", `--violet`은 "AI 제안", `--amber`는 "전문가 판단", `--rose`는 "리스크"로 사용한다.

## 슬라이드 패턴

현재 템플릿은 4장 구성이다.

### 1. Cover

용도: 강의 오프닝의 핵심 질문을 던진다.

클래스:

- `section.slide.cover.active`
- `.eyebrow`
- `.title`
- `.mark`
- `.sub`
- `.legend`
- `.pill`

수정 슬롯:

| 슬롯 | 예시 | 작성 기준 |
|---|---|---|
| 브라우저 제목 | `왜 이렇게 만들었나 - 오프닝 슬라이드` | 파일/탭 식별용 |
| eyebrow | `회계감사 AX · OPENING` | 강의명 + 세션명 |
| H1 | `왜 AI에게 다 맡기지 않았는가` | 한 문장 질문 또는 선언 |
| 강조 | `.mark` 내부 문구 | 발표자가 기억시키려는 핵심어 |
| subtitle | 코드 + AI '제안'으로 만든 이유 | H1을 설명하는 한 문장 |
| legend | 코드 -> AI 제안 -> 사람 결정 | 강의 전체의 사고 흐름 |

작성 팁:

- H1은 2줄 안에 끝낸다.
- `br`은 발표 호흡을 만들 때만 사용한다.
- `.mark`는 한 줄에 들어갈 짧은 핵심어에만 적용한다.

### 2. Three Cards

용도: 문제를 이해하기 위한 전제 3가지를 제시한다.

클래스:

- `.head`
- `.cards`
- `.card`
- `.idx`
- `.tag`

수정 슬롯:

| 슬롯 | 작성 기준 |
|---|---|
| eyebrow | `01 - 전제`, `PREMISE`처럼 번호 + 역할 |
| 제목 | `OO의 세 가지 특징` |
| 부제 | 왜 이 세 가지를 보는지 설명 |
| 카드 제목 | 각 6-10자 내외 |
| 카드 본문 | 2줄 이하 |
| 카드 태그 | 발표자가 덧붙일 맥락 |

카드 수는 3개가 가장 안정적이다. 4개 이상이면 폰트 축소보다 새 슬라이드로 나누는 편이 낫다.

### 3. Compare / Conflict

용도: 기존 요구사항과 AI 직접 사용의 충돌을 보여준다.

클래스:

- `.compare`
- `.col-h`
- `.cmp`
- `.req`
- `.bolt`
- `.llm`

수정 슬롯:

| 슬롯 | 작성 기준 |
|---|---|
| 좌측 헤더 | 기준, 요구, 현실 조건 |
| 우측 헤더 | AI 직접 사용 시 문제 |
| `.req` | 충족해야 하는 요구사항 |
| `.llm h4 em` | 실패 원인 또는 리스크 명칭 |
| `.llm p` | 짧은 설명 |

작성 팁:

- 한 행은 "요구 -> 충돌 -> 문제" 구조를 지킨다.
- 리스크명은 명사형으로 짧게 둔다.
- `--rose` 계열은 문제를 보여주는 데만 사용한다.

### 4. Role Split Flow

용도: 해결 원칙 또는 역할 분담을 제시한다.

클래스:

- `.flow`
- `.role`
- `.lab`
- `.items`
- `.note`
- `.arrow`
- `.banner`

수정 슬롯:

| 슬롯 | 작성 기준 |
|---|---|
| 역할 라벨 | 코드, AI, 사람처럼 주체를 명확히 |
| 역할 제목 | 동사형으로 작성 |
| 항목 | 실제 하는 일 2-3개 |
| note | 책임 범위 또는 제한 |
| banner | 발표자가 마지막에 읽을 결론 |

작성 팁:

- 흐름은 3단계가 가장 안정적이다.
- 각 역할 카드의 제목은 같은 문법으로 맞춘다. 예: `만든다`, `제안한다`, `결정한다`.
- 배너는 "이 강의에서 가져갈 한 문장"으로 쓴다.

## 새 슬라이드 추가 규칙

새 슬라이드는 `#deck` 안의 마지막 `section.slide` 뒤에 추가한다.

```html
<section class="slide">
  <div class="head">
    <div class="eyebrow" data-anim style="--d:0">
      <span class="dot"></span>04 - 사례 <span class="en">CASE</span>
    </div>
    <h2 class="title" data-anim style="--d:1">사례 제목</h2>
    <p class="sub" data-anim style="--d:2">사례를 보는 이유</p>
  </div>
  <!-- body -->
</section>
```

추가 후 확인할 항목:

- 새 섹션에 `slide` 클래스가 있는가
- `active`는 첫 슬라이드에만 남아 있는가
- `data-anim` 지연값이 `0, 1, 2...` 순서로 자연스러운가
- 하단의 정적 카운트 `01 / 04`를 새 총 장수에 맞게 고쳤는가
- CSS가 새 레이아웃을 지원하는가

JS는 슬라이드 개수를 자동 계산하지만, HTML 초기 로딩 순간의 정적 카운트도 맞춰두는 것이 좋다.

## 내용 교체 워크플로

1. `opening_slides_template.html`을 복사해 새 파일을 만든다.
   예: `opening_slides_tax_automation.html`

2. `<title>`과 `.chrome .brand`를 먼저 바꾼다.
   이 두 곳이 브라우저 탭과 발표 하단 브랜드에 표시된다.

3. 커버의 H1과 subtitle을 확정한다.
   이후 슬라이드는 이 질문에 답하는 구조로 쓴다.

4. 2번 슬라이드에서 전제를 3개로 정리한다.
   청중이 왜 이 문제를 중요하게 봐야 하는지 만든다.

5. 3번 슬라이드에서 충돌 또는 실패 이유를 보여준다.
   문제를 과장하지 말고, 강의 본론으로 넘어갈 이유를 만든다.

6. 4번 슬라이드에서 역할 분담 또는 해결 원칙을 제시한다.
   발표자가 실제 강의로 진입하기 전에 결론을 한 문장으로 고정한다.

7. 브라우저에서 열어 키보드 이동, 전체화면, PDF 출력을 확인한다.

## 작성 기준

- 한 슬라이드는 하나의 메시지만 가진다.
- 제목은 발표자가 그대로 읽을 수 있는 문장으로 쓴다.
- 본문은 설명문보다 구조화된 문구를 우선한다.
- 카드 본문은 2줄, 비교 설명은 1줄을 기본값으로 둔다.
- 줄바꿈은 자동에만 맡기지 말고, 발표 호흡이 필요한 지점에 `br`을 둔다.
- 숫자, 책임, 판단, 리스크처럼 오해가 생기기 쉬운 단어는 구체적으로 쓴다.
- "AI가 한다"보다 "AI가 제안한다", "사람이 결정한다"처럼 책임 주체를 분리한다.

## 수정 금지 또는 주의 항목

- `.stage`, `.deck`, `.slide`, `.active`의 기본 구조를 제거하지 않는다.
- `id="deck"`, `id="bar"`, `id="count"`, `id="dots"`는 JS가 참조하므로 바꾸지 않는다.
- `@media print`를 삭제하지 않는다.
- 슬라이드 안에 과도한 중첩 카드 구조를 만들지 않는다.
- 긴 문단을 넣기보다 장을 나눈다.
- 진행 UI가 본문을 가리지 않도록 하단 92px 안쪽에는 핵심 정보를 두지 않는다.
- 새 색상과 새 폰트를 매번 추가하지 않는다.

## 검수 체크리스트

브라우저 검수:

- `ArrowRight`, `PageDown`, `Space`로 다음 장 이동
- `ArrowLeft`, `PageUp`으로 이전 장 이동
- `Home`, `End`로 첫 장/마지막 장 이동
- `F` 또는 우측 버튼으로 전체화면 토글
- 점 네비게이션 클릭 이동
- 모바일 또는 트랙패드 환경에서 좌우 스와이프
- URL hash가 `#1`, `#2`처럼 현재 장을 반영

시각 검수:

- 모든 제목이 1280x720 기준에서 잘리지 않는가
- 하단 chrome과 본문이 겹치지 않는가
- 카드 내부 텍스트가 넘치지 않는가
- 강조색이 의미별로 일관적인가
- `data-anim` 요소가 너무 늦게 등장하지 않는가

출력 검수:

- 브라우저 인쇄 미리보기에서 각 슬라이드가 한 페이지에 들어가는가
- 진행바, 하단 chrome, 힌트가 PDF에서 숨겨지는가
- 애니메이션 전 상태가 아니라 완성 상태로 출력되는가
- 배경색과 카드 경계선이 PDF에서도 보이는가

## 재사용 요청 포맷

나중에 이 템플릿으로 새 오프닝을 만들 때는 아래 정보를 주면 된다.

```md
템플릿: lecture/opening_slides_template.html
새 파일명:
강의명:
청중:
오프닝 핵심 질문:
강의 본론으로 연결할 결론:
전제 3개:
충돌/문제 3개:
역할 분담 또는 해결 원칙 3개:
톤:
총 슬라이드 수:
반드시 포함할 표현:
피해야 할 표현:
```

정보가 부족하면 기존 4장 구조를 유지하고, `전제 -> 충돌 -> 역할 분담` 흐름으로 보수적으로 작성한다.

## Skill로 전환할 기준

현재는 이 가이드와 원본 HTML만으로 재사용하는 것이 적절하다. 다음 조건이 생기면 별도 Codex skill 생성을 검토한다.

- 같은 구조의 오프닝 HTML을 여러 강의에서 반복 생성한다.
- 입력값만 주면 HTML 파일까지 자동 생성하는 워크플로가 필요하다.
- Playwright 스크린샷 검수, PDF 출력 검수까지 자동화하고 싶다.
- 강의별로 색상, 브랜딩, 슬라이드 패턴이 늘어나 공통 규칙 관리가 필요하다.

그 전까지는 `opening_slides_template.html`을 source of truth로 두고, 이 문서를 작업 지시서로 사용한다.
