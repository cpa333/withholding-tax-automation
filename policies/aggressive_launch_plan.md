## 0. 전략 요약 (One-Pager)

**기존 보고서의 Phase 1→2→3 순차적 접근을 버리고, 리스크를 "코드로 봉인"하는 6대 안전장치를 먼저 구축한 뒤 5개 사이트 전부를 90일 이내에 동시 출시한다.**

```
T+0~30일  : 6대 안전장치 구축 (코드·계약·운영) + 3개 기관 병렬 질의 발송
T+30~60일 : Tier 1 출시 (WEHAGO + 홈택스 + 건강보험 EDI) + Tier 2 베타
T+60~90일 : Tier 2 정식 출시 (국민연금 EDI + 고용산재) — 코드 게이트로 사전허락 요건 충족
T+90일+   : 기관 회신 확보 후 마케팅 가속, 감사·컴플라이언스 리포트 차별화
```

**리스크 제어 3원칙 (Three Lines of Defense):**
1. **코드가 막는 리스크** — 인증 대리·서버 부하·허용기간 위반·우회 기술 → 코드로 원천 차단
2. **계약이 막는 리스크** — 재판매 프레임·책임 소재·BYO 원칙 → 라이선스 계약서로 명문화
3. **운영이 막는 리스크** — 약관 개정·기관 정책 변화 → 모니터링 자동화 + 서면 회신 영구 보관

## 1. 위험을 "코드로 봉인"하는 6대 안전장치 (T+0~30일 필수 구축)

> 이 장이 본 전략의 핵심이다. 각 안전장치는 단순한 권고가 아니라 **출시 게이트(Gate)**다 — 이게 없으면 어떤 Tier도 출시하지 않는다.

### 1.1 Compliance Gate 모듈 (신규) — `src/utils/compliance.py`

```python
# 의사코드 — 각 자동화 모듈이 실행 전 반드시 통과해야 하는 게이트
async def check_nps_collection_window() -> bool:
    """NPS '자료수집 허용기간' 공지를 스크래핑/캐싱하여 현재 시각이 허용기간인지 확인.
    허용기간이 아니면 자동 수집 자체를 시작하지 않음.
    → 제12조②12호 '사전 허락' 방어의 핵심 증거."""
    # 매월 1일 공단 공지 API/페이지에서 허용기간 캐싱
    # 실행 시점이 벗어나면 사용자에게 "공단 허용기간 아님 — 대기" UI 표시
```

**효과**: NPS 약관 제12조②12호 "사전 허락 없이"라는 요건에 대해 "공단이 공지한 허용기간 내에서만 작동한다"는 **코드 기반 방어 증거** 확보. 서면 회신을 기다리지 않고도 출시 가능.

### 1.2 Rate Limiter (신규) — `src/utils/rate_limiter.py`

```python
# 사이트별 RPM(분당 요청)·동시성 상한 고정 — 사용자가 변경 불가
RATE_LIMITS = {
    "wehago":    {"rpm": 30, "concurrency": 1, "min_interval_s": 2.0},
    "nhis":      {"rpm": 20, "concurrency": 1, "min_interval_s": 3.0},
    "nps":       {"rpm": 15, "concurrency": 1, "min_interval_s": 4.0},
    "comwel":    {"rpm": 20, "concurrency": 1, "min_interval_s": 3.0},
    "hometax":   {"rpm": 25, "concurrency": 1, "min_interval_s": 2.5},
}
# 업무방해죄(형법 제314조②)·약관 '정상 운영 지장' 조항 방어
```

**설계 원칙**: "사람이 손으로 하던 정도"의 속도. 사용자 설정에서 **절대 무한으로 올릴 수 없게** — 빌드 시 상수로 박아넣기. `human.py`의 `human_delay`는 이미 양호하나, 상한을 명시적 Constant로.

### 1.3 Audit Log + Compliance Report (신규) — `src/utils/audit.py`

- 모든 자동화 작업을 JSON 로그로 기록: `{"timestamp":..., "site":..., "workplace":..., "action":..., "result":..., "rate_limited":true}`
- 월간 **Compliance Report PDF 자동 생성** — 고객이 내부 감사·세무서 조사 시 "나는 규정을 준수했다"는 증거로 제출
- 이 로그는 **당사 서버로 전송하지 않음** — 고객 PC 로컬에만 저장 (BYO 원칙 준수)

**마케팅 차별화**: 다른 RPA는 없는 **"Compliance Report 자동 생성"**을 핵심 기능으로 어필. "감사 대비형 RPA" 포지셔닝.

### 1.4 `stealth.py` 재검토 — 보수적 축소 (기존 코드 수정)

현재 코드는 `navigator.webdriver=True`만 패치하고 fingerprint는 건드리지 않는 양심적 설계 (주석에 명시). 그러나 **"보안장치 우회" 프레임을 완전히 차단**하려면:

- **Option A (적극)**: `stealth.py` 자체를 삭제. `navigator.webdriver` 노출을 감수. 대신 "사람이 로그인한 실제 Chrome"임을 강조.
- **Option B (보수)**: 현재처럼 `navigator.webdriver`만 패치. 코드 주석에 "이 조치는 보안장치 우회가 아닌 Playwright 기본 마커 제거"라고 명시하고, 감사 로그에 해당 사실 기록.

**권고**: Option B. NPS 제12조②13호(CAPTCHA 우회 금지)와의 마찰을 최소화하면서도 도구의 정상성을 문서화. 단, 절대로 `webgl_vendor=True` 같은 fingerprint 스푸핑으로 확장하지 말 것.

### 1.5 Human Confirmation Gate (제출 단계 필수) — 기존 `dry_run` 확대

- 홈택스 `submit_report(ht, dry_run=True)` 구조를 **모든 제출형 작업에 적용**
- NPS·건강보험·고용산재의 "다운로드"도 민감 자료(주민번호 포함)인 경우 **다운로드 전 사용자 1회 클릭** 추가
- 전자서명법 제16조(대리 사용 금지) 취지 적극 존중 → "서명·제출은 항상 사람이 한다"는 원칙을 코드로 보장

### 1.6 No Remote Control 보장 (아키텍처 원칙)

- 절대 금지: 판매자 서버에서 고객 브라우저 원격 제어, 쿠키·세션 토큰 업로드, 자동 로그인 기능
- 현재 `chrome_cdp.py`는 `--remote-debugging-port`로 **고객 PC 로컬에서만** Chrome 연결 → 이 구조 유지
- 인증서·비밀번호는 **메모리 세션 내에서만 일시적 사용 후 폐기**, 디스크 평문 저장 금지

## 2. 5개 사이트 티어(Tier) 동시 출시 플랜

> Phase 순차 접근 대신 **위험 제어가 끝난 순서대로 티어를 열되, 전부 90일 이내에**.

### Tier 1: T+30일 출시 (법적 근거 명확, 안전장치만으로 충분)

| 사이트 | 법적 근거 | 핵심 안전장치 |
|---|---|---|
| **WEHAGO** | 자동화 금지 조항 0건 (2024-03-08 약관) | 재판매 프레임 차단(마케팅), Rate limit, BYO 로컬 |
| **홈택스** | 「홈택스 이용규정」제12조②2호 "신고서 변환 방식" **공식 합법**, 세무사법 제2조 | Human gate(제출), 접수증 자동 저장(제13조) |
| **건강보험 EDI** | 공개 약관상 자동화 금지 0건 | 업무대행기관 지정 게이트, Rate limit, 로컬 저장 |

**Tier 1 출시 조건**: 6대 안전장치 중 1.2/1.3/1.5/1.6 구축 완료 + 라이선스 계약서 1차 완성.

### Tier 2: T+60일 베타 / T+90일 정식 (코드 게이트로 조건부 허용 충족)

| 사이트 | 핵심 리스크 | 코드 기반 해소 |
|---|---|---|
| **국민연금 EDI** | 제12조②12호 "사전 허락 없이" | **1.1 Compliance Gate**로 "자료수집 허용기간" 내에서만 작동 → 공단 공지 = 사전허락의 간접 형태 주장 |
| **고용산재 토탈서비스** | 약관 미확보 | **공단 QR 자동발급 채널 우선 사용**, 화면 자동화는 보조. 7/18 점검 종료 후 약관 캡처 |

**Tier 2 출시 조건**:
- NPS: 1.1 Compliance Gate 구축 + NPS 서면 질의 발송 완료(회신 기다리지 않음) + "허용기간 외 작동 안 함"을 UI·로그로 증명
- 고용산재: 약관 원문 확보 + QR 채널 코드 우선 구현

### Tier 3: T+120일+ (확장 영역, 본 전략 범위 외)

## 3. 기관 병렬 질의 (T+0 동시 발송, 회신 기다리지 않음)

> 순차 질의가 아니라 **3개 기관에 동일한 날 발송**. 회신 전까지는 코드 게이트로 운영, 회신 도착하면 즉시 마케팅 가속.

| 대상 | 질의 채널 | 핵심 질문 | 기대 효과 |
|---|---|---|---|
| **국민연금공단** (EDI 약관 소관) | 국민신문고 + 공단 민원 | ① 정당한 권리자의 수동 로그인 후 화면 자동화 내려받기가 제12조②12호 금지인지 ② 허용된다면 조건(허용기간·빈도·건수) ③ 제12조④ 신고 목적 사용 사전승낙 필요성 | Critical 리스크 해소, Tier 2 정식 전환 |
| **근로복지공단** 고객센터 (1833-6000) | 서면 질의 | 동일 내용 + QR 자동발급 채널 활용 가이드 | 약관 미확보 리스크 제거 |
| **더존비즈온** 위하고 사업부 | 공식 파트너십 문의 | 비파트너사의 공개 UI 자동화 도구 허용 여부, 파트너십 절차 | 재판매 프레임 사전 차단 |

**첨부 서류**: 위임장, 위·수탁 계약서, "사람 로그인·저부하·로컬 저장·Compliance Gate" 설명서, 6대 안전장치 기술 문서.

**회신 활용**: 영구 보관. 향후 분쟁·고발 시 "허락받고 조건대로 했다"는 방어 근거. 부정적 회신 시 해당 모듈 즉시 비활성화 (자동 업데이트로).

## 4. 계약·마케팅 전략 (리스크 분리 + 차별화)

### 4.1 라이선스 계약서 핵심 조항 (법무 작성 전 GLM 초안)

```
제X조 (서비스의 성격)
① 본 계약은 자동화 소프트웨어의 사용허락(seat license)이며,
   당사는 고객의 WEHAGO·공단 EDI·홈택스 등 타 서비스 계정의
   대리 운영이나 자동화 수행 서비스(managed service)를 제공하지 않습니다.

제Y조 (BYO-Credentials 및 로컬 처리)
① 고객의 인증정보(공동인증서·비밀번호·OTP)와 처리 대상 데이터는
   고객의 PC에만 존재하며, 당사는 이를 수집·저장·전송하지 않습니다.
② 자동화 소프트웨어는 고객 PC 로컬에서만 실행됩니다.

제Z조 (약관 준수 책임)
① 고객은 WEHAGO·공단 EDI·홈택스 등 대상 서비스의 이용약관을
   자체 확인·준수할 책임이 있습니다.
② 자동화 소프트웨어 사용으로 인한 대상 서비스 약관 위반의 책임은
   고객에게 있습니다. 단, 당사가 해당 위반을 구조적으로 조장한 경우는 제외합니다.

제W조 (Compliance 기능)
① 소프트웨어는 사이트별 Rate Limiter, Compliance Gate, Audit Log를
   기본 제공하며, 고객은 이를 임의로 비활성화할 수 없습니다.
② 당사는 월간 Compliance Report를 고객에게 제공합니다.
```

**주의**: 면책조항(제Z조②)은 약관규제법 제8조(免责條項의 무효) 한계 내에서 작성 — 고객에게 부당하게 불리한 조항은 무효. 법무 검토 필수.

## 5. 사이트별 공격적 기능 범위 (최대 한도)

> "어디까지 갈 수 있나" — 각 사이트별로 합법선 안에서 밀어붙일 수 있는 기능의 최대치.

### 5.1 WEHAGO — **전 자동화 허용** (Phase 1 주력)

- ✅ 수임처 목록 조회
- ✅ 급여자료 반영·급여명세 PDF 생성
- ✅ 원천이행 마감
- ✅ 전자신고 파일(SWER0101) 제작
- ✅ 일괄 처리(병렬) — 단, Rate limit 내
- ⚠️ 주의: 마케팅 표현 "WEHAGO 연동 지원"

### 5.2 홈택스 — **파일변환신고 전체 + 제출은 Human Gate**

- ✅ 원천세 메뉴 진입
- ✅ 파일변환신고 업로드·검증
- ✅ 전자파일 비밀번호 팝업 입력 (사용자 확인 후)
- ✅ 제출 → 접수증 자동 저장 (제13조 준수)
- ✅ 세션 연장 (단, Rate limit 준수)
- ⚠️ 주의: **제출 버튼은 사용자 명시적 클릭** (전자서명법 제16조)

### 5.3 건강보험 EDI — **전체 다운로드 허용** (업무대행기관 전제)

- ✅ 결정내역·자격 자료 PDF/Excel 다운로드
- ✅ 사업장 전환
- ✅ 일괄 처리
- ⚠️ 주의: 업무대행기관 지정 확인을 인증 게이트로 사용

### 5.4 국민연금 EDI — **Compliance Gate 내장 조건부 전체 허용**

- ✅ 결정내역·월별 자료 다운로드 (Compliance Gate로 허용기간 강제)
- ✅ 사업장 전환
- ✅ PDF/Excel/통합저장
- ⚠️ 주의: **허용기간 외 작동 금지 UI/코드로 강제**, `stealth` 확장 금지

### 5.5 고용산재 토탈서비스 — **QR 우선 + 화면 자동화 보조**

- ✅ 공단 QR 시스템 자동발급 연동 (공식 채널) — **이게 메인**
- ✅ 보험료 부과내역(PA20209) 화면 다운로드 (보조)
- ✅ 지원금 정보 인쇄물
- ⚠️ 주의: 약관 캡처 후 진행, 사업주 동의서(개인정보보호법 제26조) 확보 게이트

## 6. 위험 시나리오 대응 플랜 (사전 준비)

| 시나리오 | 즉시 대응 | 장기 대응 |
|---|---|---|
| 특정 공단이 자동화 도구 사용 경고 발송 | 해당 모듈 자동 업데이트로 비활성화 | 서면 질의 회신 꺼내 방어 |
| 더존비즈온이 "RPA 금지" 약관 개정 공지 | WEHAGO 모듈 즉시 비활성화, 파트너십 협상 | 타 SaaS(이카운트 등) 연동 확대 |
| 고객 회계법인이 약관 위반으로 계정 정지 | 면책 조항(제Z조)로 고객 책임 | Compliance Report로 "규정 준수" 입증 지원 |
| 부정경쟁방지법 고발 | 도구 판매≠성과물 취득 방어 | 대법 2019도14725 요건 미충족 주장 |
| 언론 보도 "세무 자동화 도구 논란" | Compliance-first 마케팅 강조 | 감사 로그 투명 공개 |

**핵심**: 모든 시나리오에서 **데이터는 절대 우리 손에 없다**는 점(BYO + 로컬)이 가장 강력한 방어. 자동 업데이트로 모듈을 원격 비활성화할 수 있는 권한을 계약서로 확보해 둘 것.

## 8. 결론 — "보수적으로 분석하고, 공격적으로 실행하라"

**`glm_audit.md`가 "가능성의 지도"라면, 본 문서는 "실행의 로드맵"이다.**

- 보수적 분석이 말하는 "No-Go" 영역(국민연금 EDI 등)은 **코드 기반 안전장치로 "Conditional Go"로 바꿀 수 있다.** NPS의 "자료수집 허용기간"을 코드로 강제하면, "사전 허락 없이" 요건을 구조적으로 충족한다.
- "Phase 1→2→3 순차" 대신 **6대 안전장치를 먼저 구축하고 5개 사이트를 동시에** 연다. 안전장치가 순차적 출시의 지연 원인이 되어서는 안 된다 — 안전장치가 없으면 아무것도 출시하지 않는다는 원칙만 지킨다.
- "100% 합법 보장"은 불가능하지만, **"Compliance-first RPA"** 로 포지셔닝하면 보수 회계법인에게 오히려 강력한 차별화가 된다. 감사 로그·Compliance Report·허용기간 게이트를 마케팅 포인트로 쓴다.
- **가장 큰 방패는 "데이터 무중앙집중"**이다. 우리 서버에 고객 데이터가 1바이트도 없다는 점이 부정경쟁방지법·개인정보보호법·약관 위반 프레임 전체를 약화시킨다. 이 원칙은 절대 타협하지 말 것.

**한 문장 요약**: 6대 안전장치를 코드·계약·운영 3선으로 구축하고, 5개 사이트 전부를 90일 이내에 Compliance-first B2B RPA로 출시하라. 서면 회신은 병렬로 발송하되, 회신 기다리지 말고 코드 게이트로 사전허락 요건을 먼저 충족한다.
