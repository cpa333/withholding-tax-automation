# '원천징수자동화' GUI 프로그램 상용 판매 적법성 종합 감사 보고서

**작성**: Claude (다중 에이전트 심층 감사)
**검토 기준일**: 2026-07-17
**검토 방식**: 코드 사실관계 전수 추출(6영역) + 5개 대상 사이트 현행 약관·판례 웹 리서치(6주제) + 사이트별·판매구조별 법률 분석(6건) + 핵심 전제 적대적 검증(14건). 총 32개 전문 서브에이전트 병렬 실행.
**대상 제품**: 회계법인/세무회계사무소용 Windows 설치형 GUI(PySide6). 사용자의 로컬 Chrome에 CDP attach하여 원천세 업무를 자동화. 대상 사이트 5곳 — WEHAGO(위하고), 홈택스, 건강보험 EDI, 국민연금 EDI, 근로복지공단 토탈서비스. 라이선스는 판매자 Supabase 계정(이메일/비밀번호) 인증, 사용자 데이터는 로컬 저장.
**판매 모델**: 설치형 + 시트(인원)당 구독제로 여러 회계법인에 판매.

> **중요 고지**: 본 보고서는 공개 약관·공식 고시·판례와 실제 소스코드에 기초한 사전 리스크 감사이며, 변호사의 법률의견서를 대체하지 않습니다. 로그인 후에만 노출되는 운영정책, 개별 계약, 배포 바이너리(소스≠빌드 가능성), 기관의 개별 회신에 따라 결론이 달라질 수 있습니다. 모든 조항·판례 인용은 확인된 범위에서만 신뢰도(HIGH/MEDIUM/LOW)를 표기했습니다.

## 1. 한 줄 결론

**현재 코드 그대로의 판매는 권장하지 않는다. 그러나 이 제품은 "본질적으로 불법인 제품"이 아니라, 몇 가지 판매 전 필수 수정을 이행하면 판매 가능 범위로 들어오는 "조건부 판매 가능" 제품에 가깝다.** 단, 국민연금·건강보험 EDI 두 모듈은 예외로, 각 사이트 약관이 자동화를 **명시적으로 금지**하고 있어 별도 조치(기능 분리 또는 공식 허락 확보) 없이 판매 빌드에 포함하면 고위험이다.

핵심을 가르는 것은 "무엇을 자동화하느냐"가 아니라 **"① 탐지회피(스텔스) 코드를 탑재한 채 파느냐, ② 명시적으로 자동화를 금지한 사이트 모듈을 넣느냐, ③ 되돌릴 수 없는 신고 제출을 무인으로 하느냐, ④ 개인정보·계약 고지 없이 파느냐"** 이 네 가지다. 넷 다 코드/문서 수정으로 해소 가능하다.

## 3. 기존 policies 문서 대비 — 무엇이 바뀌었나 (가장 중요)

사용자가 policies 폴더에 모아둔 기존 검토(2026-07-08~10)는 대체로 방향이 맞지만, 이번 감사에서 **세 가지가 실측으로 뒤집히거나 보강**됐다.

### 3.1 [뒤집힘] 건강보험 EDI — "자동화 금지 조항 없음"은 이제 틀렸다
- 기존 `policies/건강보험EDI.md`는 "약관(2015-09-01)에 자동화 금지 조항이 없어 국민연금보다 유리"라고 결론냈다.
- **적대적 검증 결과 REFUTED (2026-07-17 원문 직접 확인)**: 국민건강보험공단 홈페이지 이용약관이 **2025-10-23 개정**으로 제17조 제1항 제10호에 *"공단의 사전 허락 없이 자동화된 수단을 이용하여 로그인을 시도 또는 로그인하는 행위, 공단이 제공하는 자료의 수집·처리·변경·재배포·제공하는 행위 등 … 기술적 조치를 무력화하려는 일체의 행위"* 금지를 신설했다. 같은 약관이 EDI 사업장 회원가입 시 동의받는 약관임이 공단 매뉴얼로 확인된다.
- 기존 문서가 본 것은 edi.nhis.or.kr 푸터가 여전히 링크하는 **구버전 사본**(2015-09-01)이었다.
- 뒷받침: 전자신문 2025-10-23 보도(*"건보공단·심평원·질병청이 이용약관을 개정해 스크래핑 등 자동화 데이터 추출을 제한"*), 개인정보위 2025-05-19 보도자료(약관 개정·MFA·CAPTCHA·비정상 로그인 차단 권고).
- **결론**: 건강보험 EDI는 이제 국민연금 EDI와 **동급의 명시적 금지 사이트**로 재분류해야 한다.

### 3.2 [보강] 국민연금 EDI — "사전 허락으로 해소" 계획이 실무상 작동하지 않는다
- 기존 `policies/국민연금EDI.md`는 제12조2항12호 금지를 인정하되 "**사전 허락을 받으면 이 조항이 빠진다**"를 핵심 해소책으로 삼았다.
- 검증 결과: 공단에 **자동화·스크래핑을 위한 '사전 허락'을 부여하는 공식 신청·심사·협약 절차가 확인되지 않는다**(공개 문서·업계 실태 범위). 공단이 운영하는 것은 자동화 허가가 아니라 '업무대행기관 지정'(데이터 접근 정당성)일 뿐이다. 게다가 제12호는 각 '이용자(회계법인)'의 로그인 행위를 규율하므로, 설령 허락이 있어도 **각 고객사별로 귀속**되어 판매자가 제품 차원에서 일괄 해소할 수 없다.
- 또한 2025~2026 EDI 개편으로 위하고의 '공단EDI 수집하기' 같은 상용 스크래퍼가 **이미 기술적으로 일시 중단**됐다 — 자동수집 억제 방향의 신호이자 기능 지속성 리스크.

### 3.3 [신규 발견] 스텔스/탐지회피 코드 — 기존 문서의 "보안 미우회" 전제를 코드가 정면으로 반박
- 기존 문서 전반은 "**보안장치를 우회하지 않는다 / 우회 기능 미탑재**"를 낮은 리스크의 핵심 전제로 삼았다.
- **실제 코드는 그 전제와 어긋난다** (다음 절 참조). 이것이 이번 감사의 가장 중요한 신규 발견이다.

## 4. 최대 리스크 요인: 탐지회피(안티디텍션) 코드

### 4.1 무엇이 있나 (전부 코드에서 CONFIRMED)
- `src/utils/stealth.py:16-20` — `navigator.webdriver`를 `undefined`로 재정의(은닉).
- `src/utils/stealth.py:31-55` — `playwright-stealth` 통합(`requirements.txt`에 실제 의존성, `build.py`에서 PyInstaller 번들에 포함).
- `src/utils/chrome_cdp.py:282` — Chrome 실행에 `--disable-blink-features=AutomationControlled` 플래그. 주석(279-281)에 *"NHIS EDI 보안프로그램이 navigator.webdriver를 감지해 페이지를 무한 리로드하는 것을 막는다"*고 **목적을 명시**.
- `src/automation/nhis/_doc_download.py` 및 `src/utils/nexacro.py` — MouseEvent 좌표 랜덤화(±2px)·타이밍 지터.
- `src/utils/human.py:1-4` — 모듈 docstring에 *"anti-detection … appear more natural to server-side behavioral analysis"*.
- `src/ui/workers/automation_runner.py:303` — 주석 *"수임처 간 무작위 휴식 (탐지 패턴 방지)"*.
- git 이력도 의도를 뒷받침: `534e221`("playwright-stealth 통합 + Chrome 자동화 탐지 방지"), `9396b98`("webdriver 차단 플래그 복원 + GUI stealth 적용").
- 이 스텔스는 CDP 연결 공용 경로로 **브라우저 전역 자동 적용**되어 5개 사이트 탭 전부에 적용된다(코드베이스 전역 115회 사용).

### 4.2 왜 이것이 결정적인가
이 코드는 "정당한 이용자의 공개 UI 사용"이라는 무죄 서사(대법원 2021도1533 야놀자-여기어때 계열: 서버에 **접근제한 보호조치가 없었다**는 점이 무죄의 결정적 요인)를 약화시키고, **"운영자 보호조치 무력화" 프레임**으로 재구성될 여지를 준다. 이 프레임의 종착점이 **대법원 2024도14169**(운영자가 금지·차단하는 기능을 무력화하는 프로그램의 *판매자*를 구매자와 업무방해 공동정범으로 보아 판매대금 전액 추징)이다.

- 다만 현재는 2024도14169의 전제(**명시적 금지+차단 무력화+현실적 시스템 장애+대응비용**)가 완전히 충족되진 않는다 — 사람이 직접 로그인하고, 자기 수임처 자료를 처리하며, 서버 장애가 없기 때문이다.
- **가장 위험한 조합**은 "명시적 자동화 금지 사이트(국민연금·건강보험 EDI) + 스텔스 코드 + 상용 판매"다. 이 셋이 겹칠 때 판매자의 형사·민사 책임 고리가 가장 강해진다.
- 적대적 검증(verify)의 뉘앙스: 스텔스는 "실브라우저처럼 보이게" 하는 절제형 설계이고(지문 스푸핑은 의도적으로 비활성), NHIS 플래그의 명시 동기는 "무한리로드 방지=기능 정상화"다. 이는 방어 논리의 여지이지만, "무한리로드 차단 자체가 보안프로그램의 의도적 접근제한 장치"라면 그 무력화는 오히려 정보통신망법 제48조 쪽으로 기운다 — **검증자는 이 지점을 '분석자가 유리하게 쓴 전제가 반대 결론을 지지할 수도 있는 취약점'으로 지적**했다.

### 4.3 권고
**명시적 금지 사이트(국민연금·건강보험 EDI)에는 스텔스 미적용을 원칙으로 하고, 나머지 사이트도 최소화하며, 코드·주석에서 "탐지 회피/패턴 방지" 표현을 제거하고 목적을 "서버 저부하 / 오탐으로 인한 로그인 루프 방지"로 재정의·문서화**한다. 이 조치가 "중립적 생산성 도구"라는 방어의 사활을 가른다.

## 6. 방어 가능한 사실 (제품에 유리한 확인 사항)

이 제품은 상당수의 "고위험 요소"를 이미 회피하고 있다. 아래는 모두 코드 전수 확인(grep 포함) 또는 적대적 검증으로 CONFIRMED된 사실이다.

- **대상 사이트 로그인은 사람이 직접 수행한다.** 5개 사이트 전부 ID/비밀번호/공동인증서를 프로그램이 자동 입력·저장하지 않고, 사람이 로그인한 세션에 CDP attach만 한다(`wait_for_login`은 DOM 폴링으로 완료만 감지). 코드에 "Human-in-the-loop" 설계가 명시됨. → 정보통신망법 침입죄의 "이용자 직접 인증" 방어와 대법원 2005도870의 '이용자 도구' 구성에 유리.
- **공동인증서 비밀번호·전자서명을 일절 취급하지 않는다.** 유일하게 다루는 비밀번호는 '전자신고 변환파일 비밀번호'로, 매 실행 시 사용자가 GUI에 입력하고 저장·외부전송 없이 홈택스/위하고 폼에 주입만 된다.
- **업무 데이터(수임처·급여·주민번호)를 외부로 전송하지 않는다.** 전수 grep 결과 런타임 외부 통신은 ①Supabase 인증(이메일/비번/토큰만), ②GitHub 업데이트(GET 전용), ③대상 사이트 5곳(사용자 로컬 브라우저 경유)뿐. **텔레메트리·Sentry·analytics·화면캡처 업로드 코드 0건.** → 판매자의 개인정보 '수탁자' 지위 회피 가능, 부정경쟁 성과도용의 '대상' 부존재.
- **판매자 인프라는 저부하 단발 요청이다.** 인증 재검증 4시간 주기, 업데이트 확인 20시간 스로틀, 재시도 루프 없음.
- **홈택스는 데이터 수집 코드 자체가 없다**(`_download.py` 부재). 파일 업로드→검증→제출만 수행.
- **근로복지공단은 순수 다운로드(read-only)** — 제출·신고·발급 등 대외효과 동작 없음.
- **세무사법 리스크는 낮다.** 삼쩜삼(자비스앤빌런즈) 세무사법 위반 논란이 **최종 무혐의**(2025-06 대검 재항고 기각)로 종결됐고, 본건은 무자격 B2C가 아니라 **자격자(회계법인/세무사) 대상 B2B 도구 판매**여서 '세무대리' 요건을 충족하기 어렵다. 소프트웨어 판매 자체는 세무사법 제20조가 금지하는 대상이 아니다.
- **로컬 실행형 자동화 판매자 무죄 판례가 있다.** 대법원 2017도16520(매크로 판매자 무죄), 2019도2862(로컬 실행·서버 무변경은 악성프로그램 아님) — 스텔스만 제거하면 이 계열에 안착한다.

## 7. 판매 전 필수 조치 체크리스트

### 7.1 필수 (판매 전 반드시 — 이걸 안 하면 판매 권장 불가)
1. **탐지회피 코드 정리**: 명시적 금지 사이트(국민연금·건강보험 EDI)에 스텔스 미적용, 나머지 최소화, 주석에서 "탐지 회피" 의도 제거·목적 재문서화. `WTAX_NO_DELAY` 지연무력화 토글을 릴리스 빌드에서 완전 제거.
2. **국민연금·건강보험 EDI 모듈 처리**: 공식 허락/연계 경로 확보 전까지 판매 SKU에서 **분리 또는 기본 비활성화**. (두 사이트 모두 자동화를 명시적으로 금지. 국민연금은 개편으로 이미 기술 중단 상태라 제외 시 제품 손실도 작음.)
3. **홈택스 무인 제출 제거**: dry-run 해제 후에도 대상 수임처·건수·세액 요약을 보여주고 사람이 명시적으로 확인해야만 최종 '제출' 클릭이 실행되도록 코드 게이트 강제. 확인모달 자동클릭 삭제.
4. **홈택스 세션타이머 조작 함수 프로덕션 제외** (`trigger_session_popup_soon`).
5. **개인정보처리방침 작성·게시 + 국외이전 고지**: 앱 계정 이메일/비밀번호가 Supabase 국외 서버로 전송·보관됨을 국가·항목·목적·방법·거부권과 함께 공개하거나 별도 동의. **Supabase 프로젝트 실제 리전을 대시보드에서 확인**할 것(외부에서 미확인).

8. **보험료납부확인서 발급 dead 배선 완전 제거.**
9. **구매자 자격 한정(온보딩/계약)**: 회계법인/세무법인(또는 요건 충족 세무사) + 각 공단 업무대행기관 지정 보유 확인. 개인 공인회계사 사무소는 고용·산재 대행 자격이 없으므로(헌재 2020헌마139) 근복 모듈 판매 대상에서 제외.

## 9. 확인하지 못한 것 (잔존 불확실성 — 판매 전 실물 확인 권장)

정직하게, 다음은 이번 감사에서 확인하지 못했고 결과가 바뀔 수 있는 부분이다.

1. **로그인 후에만 노출되는 운영정책/개별 약관** — 위하고 v13 약관이 인용하는 별도 '서비스 이용정책', 홈택스 회원가입 화면 '이용약관 동의' 전문(WebSquare 동적 렌더링), 건강보험/국민연금/근복 EDI 로그인 후 화면의 추가 동의문. 인증장벽으로 미열람. **여기에 자동화 금지가 있으면 해당 사이트 등급이 상향된다.**
2. **근로복지공단 토탈서비스 웹 「사이트 이용약관」** — 조사 시점 사이트 전면점검(7.16~18) + SPA로 미확인. 점검 종료 후 실물 확인 필수.
3. **Supabase 실제 호스팅 리전** — Cloudflare 뒤라 외부에서 미확인. 국외이전 고지 의무 성립 요건.
4. **배포 바이너리 = 소스코드** 여부 — 본 감사는 레포 소스 기준. 실제 릴리스 .exe가 이 소스에서 빌드되는지(숨은 자격증명 자동입력·데이터 전송 경로 부재)는 별도 검증(빌드 재현 또는 mitmproxy 네트워크 캡처) 권장.
5. **각 기관의 '판매자' 제재 전례** — 사업장 EDI 업무 자동화 '판매자'를 수사·기소한 사례는 확인되지 않았으나(대체로 개인 의료데이터 수집 앱이 규제 대상), 부존재를 단정할 수는 없음.

## 10. 종합 의견

**객관적·현실적으로, 이 제품은 "팔 수 없는 제품"이 아니다.** 인프라는 깨끗하고(업무데이터 외부 전송 없음, 텔레메트리 없음, 로컬 저장), 대상 사이트 로그인을 자동화하지 않으며, 세무사법 리스크도 낮고, 홈택스·근로복지공단은 오히려 공식 경로/read-only로 안전한 축에 든다. 형사 침입·업무방해·악성프로그램의 무죄 판례 계열에 상당히 근접해 있다.

**그러나 현재 코드 그대로의 판매는 권장하지 않는다.** 판매(자기사용과 달리)는 판매자에게 '2차 책임층'을 얹는데, 이 제품에는 그 책임을 실제로 활성화시킬 수 있는 요소 — ①전역 탐지회피 코드, ②명시적으로 자동화를 금지한 두 EDI 모듈, ③홈택스 무인 제출, ④개인정보·계약 고지 공백 — 이 동시에 존재한다. 특히 "명시적 금지 사이트 + 스텔스 + 상용 판매"의 조합은 대법원 2024도14169(무력화 도구 판매자 유죄+추징)로 가는 유일한 실질 통로다.

**현실적 로드맵**:
- **지금 당장 팔 수 있는 최소 구성**: 홈택스(무인 제출 제거) + 근로복지공단(약관 실물 확인 후) + 위하고 모듈만으로 스텔스를 걷어내고, 개인정보처리방침·EULA를 갖춰 자격자 대상 B2B로 판매. 이 구성이 리스크 대비 사업성이 가장 낫다.
- **국민연금·건강보험 EDI**는 별도 트랙으로 분리해 공식 허락/연계 경로를 타진하고, 확보 전에는 기본 비활성화. 자동수집 규제가 강화되는 방향(개인정보위·공단 약관 개정)이라 이 부분은 장기적으로 스크래핑 의존 자체를 줄이는 게 안전하다.

가장 큰 실질 리스크는 형사 처벌이 아니라 **①홈택스 무인 제출로 인한 오신고 가산세 손해배상**과 **②상류기관(더존·공단·국세청)의 약관 개정·기술 차단으로 인한 제품 지속성**이다. 전자는 사람 최종 확인 게이트로, 후자는 원격 기능 비활성화 설계와 계약상 고지·SLA로 관리해야 한다.
