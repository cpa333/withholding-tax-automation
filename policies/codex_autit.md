## 1. 최종 결론

### 1.1 현재 코드 전체를 그대로 유상 판매하는 것은 **NO-GO(출시 보류)**

프로그램의 판매행위 자체가 곧바로 범죄라고 단정할 수는 없다. 그러나 현재 구현과 판매 범위를 합쳐 보면, 여러 회계법인에 전 기능을 즉시 판매하기에는 명확한 계약상 금지, 개인정보·보안 결함, 되돌리기 어려운 자동 행위, 오픈소스 라이선스 문제가 동시에 존재한다.

가장 중요한 출시 차단 사유는 다음과 같다.

1. **국민건강보험공단 최신 약관이 자동화를 명시적으로 금지한다.** 2025-10-23 시행 약관 제17조는 공단의 사전 허락 없는 자동화 로그인, 자료의 수집·처리·변경·재배포·제공 및 자동화를 막기 위한 기술적 조치의 무력화를 금지한다. 기존 `건강보험EDI.md`와 `grok_autit.md`가 전제한 2015년판 결론은 현재 기준으로 폐기해야 한다.
2. **국민연금 EDI 약관도 사전 허락 없는 자동화 로그인을 금지하고, 로그인 후 인증정보를 이용한 개인정보 수집까지 명시한다.** 사람에게 로그인을 맡긴다는 사정만으로 해결되지 않는다.
3. **코드가 자동화 사실을 의도적으로 숨긴다.** `navigator.webdriver`를 숨기고 `--disable-blink-features=AutomationControlled`와 `playwright-stealth`를 사용한다. CAPTCHA 풀이·IP 순환은 없지만, “보안조치나 탐지를 우회하지 않는다”는 기존 검토 전제와 맞지 않는다.
4. **홈택스에서 실제 신고를 자동 완료할 수 있다.** dry-run을 해제하면 최종 제출 버튼뿐 아니라 두 차례 확인창까지 자동으로 누른다. 기존 문서의 “최종 제출은 사람이 한다”는 전제와 실제 코드가 다르다.
5. **국민연금 원본의 주민등록번호 전체표출 자료를 내려받아 평문으로 장기 보관한다.** 현재 업무 계산에는 주민등록번호 전체가 필요하지 않는데도 원본 Excel/PDF가 남고 자동 파기·암호화 정책이 없다.
6. **공단 자료가 로컬에만 머물지 않는다.** NHIS·NPS·COMWEL 자료를 근로자명으로 결합하여 WEHAGO 급여 Excel에 기록한 뒤 WEHAGO 서버에 업로드한다. 기존 자료의 “외부 전송 없음”은 기술적 사실과 다르다.
7. **`PyMuPDF`가 상용 설치본에 의도적으로 포함된다.** PyMuPDF는 AGPL 또는 상용 라이선스 체계이다. 폐쇄형 유료 제품으로 배포하면서 AGPL 의무를 이행하지 않는다면 별도의 저작권 리스크가 된다.
8. **판매용 제품·계약·보안 통제가 아직 준비되지 않았다.** 시트 수 통제, 조직·사용자 관리, EULA, 개인정보처리방침, 위수탁 부속서, 보유·파기, 암호화, 코드서명, 정확한 배포자 표시가 없다.

### 1.2 다만, 구조를 바꾸면 **조건부 판매는 현실적으로 가능**

가장 빠르고 보수적인 상용화 경로는 포털 자동화를 먼저 파는 것이 아니라 다음 범위의 **로컬 문서처리 SKU**를 분리하는 것이다.

- 사용자가 각 포털에서 정식으로 직접 내려받은 파일을 수동으로 선택
- 로컬에서 보험료·급여자료를 병합·검증·변환
- 업로드 전 변경 내역을 사람에게 표시하고 승인받음
- 홈택스 최종 제출은 홈택스의 원래 화면에서 사용자가 직접 수행
- 공급자는 고객의 인증정보·세무자료·주민등록번호에 접근하지 않음

이 범위도 PyMuPDF, 개인정보 설계, 계약, 코드서명 문제를 해결해야 하지만, 공단 약관의 자동화 금지와 정면으로 충돌하는 핵심 위험은 크게 줄어든다.

포털 자동화까지 판매하려면 최소한 아래 조건이 충족되어야 한다.

- NHIS와 NPS로부터 해당 시나리오를 구체적으로 설명한 **사전 서면 허락** 확보
- COMWEL 최신 약관 확보 및 필요 시 서면 허락
- WEHAGO와 국세청에 상용 로컬 RPA의 허용 범위·속도·최종 제출 방식 서면 확인
- stealth와 자동화 탐지 은닉 제거
- 주민등록번호 전체표출 제거, 최소수집·암호화·자동파기 구현
- 홈택스 자동 최종제출 및 WEHAGO의 파괴적 변경에 강제적 사람 승인 추가
- 오픈소스 라이선스와 판매용 계약·개인정보·보안 체계 완비

## 2. 이 보고서가 구분하는 세 가지 문제

“합법인가?”는 하나의 질문처럼 보이지만 실제로는 다음 세 층위가 다르다.

| 층위 | 판단 |
|---|---|
| 프로그램을 제작·판매하는 행위 | 일반적으로 곧바로 금지되는 것은 아님. 다만 기능·광고·지원 방식에 따라 무자격 세무대리, 개인정보 처리, 저작권 침해 등의 문제가 생길 수 있음 |
| 고객이 포털에서 프로그램을 사용하는 행위 | 사이트 약관과 계정 권한의 적용 대상. 현재 NHIS·NPS 자동화는 명시 조항과 충돌할 가능성이 매우 높음 |
| 형사책임 | 약관 위반만으로 자동 성립하지 않음. 접근권한 초과, 기술적 제한 회피, 타인의 식별부호 이용, 실제 시스템 장애 등 별도 요건이 필요 |

따라서 “약관 위반 가능성”을 곧바로 “범죄”라고 표현해서도 안 되고, 반대로 “형사 유죄 가능성이 높지 않다”는 이유로 유상 판매가 안전하다고 결론내려서도 안 된다. 다수 고객에게 판매하는 사업에서는 계정정지, 신고 지연, 고객 손해배상, 기관의 중단 요구, 평판 및 유지보수 중단이 더 현실적인 1차 리스크다.

## 3. 조사 방법과 증거 수준

이번 감사는 다음 자료를 서로 대조했다.

- 프로젝트 전체 정적 소스 감사
- `policies/건강보험EDI.md`, `국민연금EDI.md`, `위하고.md`, WEHAGO Word 보고서, `grok_autit.md`

- PyMuPDF·Qt의 공식 라이선스 문서

- **확인**: 코드 또는 공식 원문에서 직접 확인
- **해석**: 확인된 사실을 법적·사업적으로 평가한 의견
- **미확인**: 공식 원문에 접근하지 못했거나 로그인 후 개별 계약을 추가로 확인해야 함

정적 감사이므로 실제 배포 바이너리의 패킷, 고객별 WEHAGO 계약, 로그인 후 공지, 런타임에서 동적으로 내려오는 스크립트는 별도 검증 대상이다.

### 4.1 개인정보·인증 데이터

- NPS 자동화는 “전체표출”을 선택한다. 주석과 파서가 원본에 성명·주민등록번호가 포함됨을 전제로 한다: `src/automation/nps/_constants.py:63`, `src/utils/raw_data_reader.py:144`.
- 기본 저장 위치는 `공단EDI_YYYYMM/수임처/포털`이며, 실패 시 문서·홈·LocalAppData·TEMP로 폴백한다: `src/utils/save_path.py:67`.
- 작업 완료 후 원본을 보유기간에 따라 일괄 파기하는 기능이 없다.
- SQLite에는 수임처명, 사업자등록번호, 관리번호, 신고주기, 오류·traceback·출력경로가 저장된다: `src/batch/db.py:35`.
- 제거 프로그램은 사용자 데이터와 결과를 남겨 둔다: `installer.iss:122`.
- Supabase Auth로 프로그램 계정의 이메일·비밀번호가 전송되고 access/refresh token과 user 객체가 `auth_session.json`에 평문 저장된다: `src/utils/auth.py:48`, `src/utils/auth.py:103`, `src/utils/auth.py:133`.
- 포털 비밀번호나 공동인증서를 코드가 직접 저장하는 사실은 찾지 못했다. 이는 완화 요소다.
- `USER_GUIDE.md`와 `tools/add_defender_exclusion.bat`은 Defender 오탐을 이유로 예외 추가를 안내한다. 다수 고객에게 배포할 제품에서 광범위한 보안 예외를 표준 설치절차로 요구하는 것은 공급망·감사·고객 신뢰 측면에서 부적절하다. 코드서명과 오탐 해소를 우선해야 한다.

### 4.2 자동화 은닉과 내부 객체 제어

- `src/utils/chrome_cdp.py:278`은 NHIS 보안프로그램의 `navigator.webdriver` 감지를 피한다는 주석과 함께 `--disable-blink-features=AutomationControlled`를 사용한다.
- `src/utils/stealth.py:1`은 webdriver, plugins, permissions, vendor, Chrome 객체 등 자동화 흔적을 숨긴다.
- `src/utils/human.py:1-16`은 지연의 목적을 서버 측 행동분석에 자연스럽게 보이기 위한 anti-detection이라고 직접 설명한다. 부하 제한을 위한 정직한 rate limit과 탐지 회피를 위한 인간 흉내는 구분해야 한다.
- NHIS는 Nexacro 내부 객체와 이벤트를 직접 호출하고 `pageLinkPopup1('201')` 같은 사이트 함수를 사용한다: `src/automation/nhis/_nexacro.py:30`, `src/automation/nhis/_doc_access.py:28`.
- WEHAGO는 RealGrid 전역 객체, React 내부 인스턴스·`setState`, `window.open` 가로채기를 사용한다: `src/automation/wehago/_swsa_excel.py:20`, `src/automation/wehago/_swsa_constants.py:42`, `src/automation/wehago/_common.py:823`.
- 홈택스 CLI 경로에는 내부 `$c.pp.sessionXtn($p)`와 `sessionTimer('Y')`를 호출하는 세션 연장이 있다: `src/automation/hometax/_session.py:15`. 일반 GUI Phase 10에서는 현재 시작되지 않더라도 상용 배포 코드에 존재한다.
- CAPTCHA 해제 서비스, 프록시·VPN·IP 순환, 포털 비공개 HTTP API 직접 호출은 찾지 못했다. 그러나 이것이 webdriver 은닉을 없던 일로 만들지는 않는다.

## 5. 기존 `policies/` 판단의 중대한 정정

| 기존 전제 | 2026-07-17 확인 결과 |
|---|---|
| NHIS 약관은 2015년판이고 자동화 금지가 없음 | **오류.** 현재 공식 페이지에는 2025-10-23 시행본이 있고 제17조에 사전 허락 없는 자동화·자료수집·기술조치 무력화 금지가 있음 |
| 공단 자료는 로컬에서만 처리되고 외부 전송 없음 | **오류.** 공단 값을 WEHAGO 업로드 Excel에 결합해 전송 |
| 필요한 정보만 수집 | **오류.** 사용하지 않는 주민등록번호 전체표출 원본까지 저장 |
| 임시파일 자동 삭제 | **오류.** 자동 보유기간·일괄 파기 기능을 찾지 못함 |
| WEHAGO는 입력 중심이고 정보를 수집하지 않음 | **불완전.** 수임처 목록·급여 Excel·PDF도 내려받음 |
| 자동화 탐지 회피 없음 | **오류.** webdriver 은닉과 stealth가 명시적으로 존재 |
| 홈택스 최종 제출은 사람이 함 | **오류.** dry-run 해제 시 제출 및 확인창까지 자동 수행 |
| 시트 구독 판매 준비가 됨 | **오류.** 현재는 베타 계정 게이트이며 좌석·조직·결제·동시접속 통제가 없음 |

따라서 기존 자료는 참고 이력으로만 보존하고, 출시 의사결정에는 이 보고서와 최신 원문을 사용해야 한다.

### 6.1 국민건강보험공단 EDI — **STOP, 사전 허락 전 판매 제외**

[국민건강보험공단 현재 이용약관](https://www.nhis.or.kr/nhis/etc/wbhaze01900m01.do)은 EDI를 포함한 공단 사이트 서비스에 적용되는 공개 약관이며, 최신 표시본은 2025-10-23 시행이다.

제17조는 공단의 사전 허락 없이 다음을 하는 것을 금지한다.

- 공단 자료를 수집·처리·변경·재배포·제공하는 등 실제 이용을 전제로 한 서비스 취지에 맞지 않는 방식의 이용

공단은 별도의 [EDI 업무대행 절차](https://edi.nhis.or.kr/homeapp/wep/o/agencyBusiness.do)와 [업무대행기관 가입 절차](https://edi.nhis.or.kr/homeapp/wep/o/InsuNotAdptSignUp.do)를 운영한다. 고객 회계법인이 이 절차와 개별 사업장 위임을 갖추는 것은 필수지만, 업무를 대행할 자격·권한과 브라우저 자동화에 대한 제17조상 사전허락은 별개의 요건이다.

**판정:** NHIS 자동화 모듈은 상용 빌드에서 하드 비활성화하고, 공단이 구체적 자동화 방식과 상용 배포를 알고 승인한 서면을 확보한 뒤에만 활성화한다. 저속 실행·사람 로그인·정식 업무대행 지정은 필요조건이지만 명시적 “사전 허락”을 대신하지 않는다.

### 6.2 국민연금 EDI — **STOP, 사전 허락 전 판매 제외**

[국민연금 EDI 이용약관](https://edi.nps.or.kr/)은 2024-08-01 시행본을 현재 홈페이지에서 직접 제공한다.

- 제12조 제2항 제12호: 사전 허락 없이 자동화 수단으로 로그인 시도 또는 로그인 금지. 괄호에서 **인증정보를 이용하여 로그인한 뒤 개인정보를 수집하는 행위까지 포함**
- 제12조 제2항 제13호: IP 지속 변경과 CAPTCHA 외부 우회·무력화 금지

사람이 최초 로그인을 한다는 점은 긍정적이지만, 약관이 로그인 후 개인정보 수집까지 명시했으므로 핵심 충돌을 제거하지 못한다. 전체표출 주민등록번호 자료를 자동 다운로드하고 WEHAGO 입력에 재사용하는 현재 흐름은 더욱 불리하다.

**판정:** NPS 자동화는 서면 허락 전 상용 제품에서 제거한다. 허락을 받더라도 전체표출이 정말 필요한지 별도로 승인받고, 기본은 마스킹·최소필드로 바꾼다.

### 6.3 근로복지공단 고용·산재보험 토탈서비스 — **HOLD**

[고용·산재보험 토탈서비스](https://total.comwel.or.kr/)는 조사 시점에 2026-07-16 20:00부터 2026-07-18 24:00까지의 점검 안내만 반환해 최신 이용약관 원문을 확인할 수 없었다. 검색 결과나 타사 관행은 공식 허락을 대신하지 않는다.

코드는 고용·산재 부과자료와 지원금 자료를 자동으로 내려받고, 특정 사무대행 사업자번호를 안내문에 하드코딩한다: `src/automation/comwel/_common.py:120`. 고객마다 보험사무대행기관 지정·위임과 사용자 권한이 실제로 있는지 검증해야 한다.

[고용산재보험료징수법 제33조](https://www.law.go.kr/LSW/lsLawLinkInfo.do?chrClsCd=010202&lsJoLnkSeq=900112421)와 [현행 시행령 제44조](https://www.law.go.kr/LSW/lsLawLinkInfo.do?chrClsCd=010202&lsJoLnkSeq=900112437)는 공단 인가를 받은 보험사무대행기관과 사업주의 위임을 전제로 한다. 관계 법률에 따라 등록된 법인은 자격 경로가 될 수 있지만, 모든 회계법인·모든 개인 CPA가 아무 절차 없이 자동으로 토탈서비스 대행권을 얻는다는 뜻은 아니다. 고객 법인의 공단 인가와 각 사업주의 수임·위임을 별도로 확인해야 한다.

### 6.4 WEHAGO — **현재 코드 NO-GO, 축소·개선 후 제한적 Conditional Go**

[WEHAGO 서비스 이용약관](https://www.wehago.com/policy.html?key=65)은 현재 공개된 제13판, 2024-03-08 시행본이다. 매크로·RPA·봇을 이름으로 포괄 금지하는 문구는 확인하지 못했다. 다만 이것은 “허용을 확인했다”는 뜻이 아니다. 제3조 제5항은 개별 서비스 약관을 우선시키고, 제5조 제1항은 계정의 제3자 사용을 제한한다. 제8조 제1항 제7호는 회사가 제공하는 목적 외 영리적 복사·복제·재판매·재임대·양도를 금지하고, 제9조는 전자적 침해·계약위반으로 서비스에 중대한 지장 또는 그 우려가 있을 때 이용을 제한할 수 있게 한다. 제10조 제4항은 세무·회계 등 업무정보를 계약 이행 목적으로만 사용하도록 하고 제28조는 이용권·계약상 지위의 양도·판매를 제한한다. 고객별 주문서·특약도 추가 확인 대상이다.

독립 소프트웨어를 판매하는 것이 곧 WEHAGO 서비스 자체를 재판매하는 것은 아니므로 “재판매 금지” 조항만으로 제품 판매가 당연히 불법이라고 단정하기 어렵다. 반면 현재 구현은 다음 이유로 단순 공개 UI 보조보다 위험하다.

- 수임처와 급여자료를 자동 수집
- RealGrid·React 내부 객체를 직접 조작
- 기존 급여자료를 “삭제 후 업로드”로 대체
- 원천세 마감해제·재마감을 사람의 별도 최종 승인 없이 수행
- 전역 stealth로 자동화 흔적을 숨김
- 공단에서 받은 개인정보를 WEHAGO로 재업로드

**판정:** stealth와 내부 객체 의존을 제거하고, 공개 UI 또는 공식 API·가져오기 기능만 사용하며, 변경 전 diff·백업·개별 승인·복구 절차를 만든 WEHAGO-only 파일럿은 법률검토 후 조건부로 가능하다. 여러 법인에 유료 확장하기 전에는 더존에 “사용자가 직접 로그인한 로컬 설치형 상용 RPA”의 허용 여부와 속도·계정·지원 조건을 서면 확인하는 것이 현실적으로 필요하다. 공식 제휴·승인이라는 표현은 회신이나 계약 없이는 사용하면 안 된다.

조사일 현재 공개된 정상 작동 개발자 API 문서와 API 이용약관은 찾지 못했다. 비공개 파트너 경로가 있을 수 있으므로 [WEHAGO T 공식 문의](https://www.wehago.com/common/inquire/tinquire.html)를 통해 확인한다.

### 6.5 홈택스 — **업로드·검증은 개선 후 Conditional, 자동 최종 제출은 STOP**

[홈택스 공개 이용약관](https://hometax.go.kr/ui/pp/a/b/UTXPPABA23.xml)은 현재 공식 페이지가 제공하는 약관이다. 제15조는 사전승낙 없는 서비스 이용 영리행위, 얻은 정보의 복사·변경·타인 제공, 안정적 운영에 지장을 줄 우려가 있는 행위를 제한한다. 공개 약관에는 매크로라는 단어의 포괄 금지는 확인되지 않았다.

[현행 「홈택스 이용에 관한 규정」](https://www.law.go.kr/LSW/admRulLsInfoP.do?admRulSeq=2100000274724)은 국세청고시 제2026-3호로 2026-02-18 시행되었다.

- 제9조는 납세자·세무대리인·세법상 신고대행 가능자를 전자신고 이용대상으로 규정한다.
- 제10조는 원천징수이행상황신고를 전자신고 대상으로 둔다.
- 제12조는 신고서 작성, 신고서 변환, 신고서 확인 방식을 인정한다.
- 제13조는 정상 전송·접수 여부 확인과 접수증 보관을 요구한다.

따라서 정당한 세무대리인이 `.01` 파일을 변환신고 방식으로 올리는 업무 목적 자체는 제도와 맞는다. 그러나 이 규정은 제3자 상용 RPA나 자동 최종 제출을 승인한 문서가 아니다.

국세청은 2025-05-27 [세무플랫폼 대응 정보화전략계획 수립 사업 공고](https://sc.nts.go.kr/nts/na/ntt/selectNttInfo.do?mi=2207&nttSn=1343162)에서 홈택스 스크래핑에 대한 법·제도·기술적 종합 대응전략과, 납세자 동의를 전제로 세무플랫폼에 국세자료를 제공하고 이용내역·수수료를 관리하는 Open API 시스템의 **설계**를 과업으로 제시했다. 이는 모든 자동화를 금지했다는 공고도 아니고, 공개 API가 이미 운영된다는 허가도 아니다. 다만 무승인 스크래핑보다 승인형 연계 경로를 선호하는 정책 방향을 보여 주므로, 상용 RPA를 서면 확인 없이 확대하는 데 불리한 신호다. 현재 홈택스 모듈은 자료수집형 스크래핑보다 신고파일 업로드·제출에 가깝다는 차이도 함께 고려해야 한다.

현재 `src/automation/hometax/_upload.py:247-293`은 dry-run 해제 시 “전자파일 제출하기”와 두 확인창을 자동 클릭하고 접수결과까지 기다린다. 하나의 전역 dry-run 체크박스를 해제한 사실만으로 각 수임처 신고에 대한 충분한 최종 의사확인이라고 보기 어렵고, 오류·중복·잘못된 귀속기간·가산세 손해가 발생할 수 있다.

**판정:**

- 파일 선택·형식 검증·오류 표시까지: 사람 로그인, 저부하, 공식 화면 사용, 개인정보 통제 후 조건부 가능
- 제출 직전 요약 화면: 납세자, 귀속기간, 지급월, 세액, 파일 해시, 중복 신고 여부를 표시
- 최종 제출: 홈택스 원래 화면에서 권한 있는 담당자가 직접 클릭하도록 상용 자동화에서 제거
- 내부 세션연장 함수: 제거
- 국세청 서면 회신이나 공식 연계 없이 “홈택스 공식 연동·승인” 표현 금지

## 7. 관련 법률·판례의 현실적 적용

### 7.1 정보통신망 접근

[정보통신망법 제48조 제1항](https://www.law.go.kr/lsLinkCommonInfo.do?chrClsCd=010202&lsJoLnkSeq=1028660557)은 정당한 접근권한 없이 또는 허용된 접근권한을 넘어 정보통신망에 침입하는 행위를 금지한다.

[대법원 2022. 5. 12. 선고 2021도1533](https://file.scourt.go.kr/dcboard/1727143941880_111221.pdf)은 접근권한 제한 여부를 보호조치·이용약관 등 객관적 사정으로 신중하게 판단해야 한다고 했다. 그 사건에서는 정보의 영리 이용 제한이 접근 자체의 제한과 같지 않고, 실제 접근 보호조치가 없었던 점 때문에 침입죄가 부정되었다. 또한 컴퓨터등장애업무방해는 정보처리 장애가 현실적으로 발생해야 한다고 봤다.

이 판결은 “모든 브라우저 자동화는 합법”이라는 면허가 아니다. 현재 NHIS·NPS에는 자동화 로그인·수집을 직접 겨냥한 최신 약관이 있고, 코드에는 자동화 탐지 은닉이 있다. 이는 2021도1533보다 불리한 객관적 사정이다.

[대법원 2024. 11. 14. 선고 2021도5555](https://www.law.go.kr/LSW/precInfoP.do?precSeq=599601)는 서비스제공자가 접근권한 범위를 정하며, 비밀번호를 새로 입력하지 않고 이미 로그인된 상태를 이용했더라도 서비스제공자의 의사에 반한 제3자 접근은 침입이 될 수 있다고 보았다. 따라서 고객 직원의 정당한 계정·권한, 본인 조작, 명확한 위임을 유지하고 공급자가 원격으로 고객 계정에 접속하는 운영모델은 피해야 한다.

### 7.2 컴퓨터등장애업무방해

형법 제314조 제2항은 부정한 명령 등으로 정보처리에 실제 장애를 발생시켜 업무를 방해한 경우를 처벌한다. 느린 속도와 적은 요청량은 위험을 낮추지만, 여러 회계법인에 판매하면 총 요청량이 커진다. 고객별 병렬 3세션, 무제한 수임처 순회, 재시도 폭주는 중앙 통제가 없더라도 기관 관점에서 누적될 수 있다.

제품에 다음 하드 제한이 필요하다.

- 포털별 동시성 1 또는 기관 승인 수치
- 최소 요청 간격과 지수 백오프
- 401·403·429, CAPTCHA, 반복 오류·차단 문구 감지 시 즉시 중단
- 원격 kill switch는 최소 정보만 사용하고 오남용 방지
- 기관별 전체 고객 사용량을 추정할 수 있는 익명화 지표 또는 보수적 기본값. 단, 개인정보·업무자료 텔레메트리는 전송하지 않음

### 7.3 개인정보 보호

[개인정보 보호법 제26조](https://law.go.kr/lsLawLinkInfo.do?chrClsCd=010202&lsJoLnkSeq=900079876)는 처리업무 위탁 시 목적 외 처리 금지, 안전조치, 공개, 수탁자 감독 등을 요구한다. 공급자가 단순히 로컬 도구만 제공하고 고객 자료에 접근하지 않는다면 고객 회계법인이 주된 개인정보처리자가 될 가능성이 크다. 그러나 원격지원, 오류파일 업로드, 화면 캡처, 중앙 로그, 클라우드 백업을 통해 공급자가 자료를 볼 수 있다면 수탁자 또는 별도 처리자 지위가 생길 수 있다.

[제24조의2](https://www.law.go.kr/lsLinkCommonInfo.do?lsJoLnkSeq=1006184231)는 주민등록번호 처리를 법령상 근거가 있는 경우 등으로 제한한다. 고객의 세무·사회보험 업무에 법적 처리근거가 있더라도, 프로그램이 실제 계산에 필요하지 않은 전체 번호를 일률적으로 수집·보관하는 것은 최소수집과 안전성 측면에서 방어하기 어렵다.

[제29조](https://www.law.go.kr/LSW/lsLawLinkInfo.do?chrClsCd=010202&lsJoLnkSeq=900079357)는 기술적·관리적·물리적 안전조치를 요구하고, [제21조](https://www.law.go.kr/LSW/lsLinkCommonInfo.do?ancYnChk=&chrClsCd=010202&lsJoLnkSeq=1020398651)는 목적 달성·보유기간 경과 후 복구되지 않도록 파기하도록 한다. 현재 평문 파일, 평문 토큰, 무기한 결과 보존, 제거 후 잔존은 상용 제품의 기본선에 미달한다.

Supabase의 실제 리전이나 지원 과정에서 국외 접근이 발생한다면 [제28조의8](https://www.law.go.kr/LSW/lsInfoP.do?ancNo=19234&ancYd=20230314&ancYnChk=0&chrClsCd=010202&efGubun=Y&efYd=20240315&lsiSeq=248613&nwJoYnInfo=Y)에 따른 국외 이전 근거·고지·보호조치를 검토해야 한다. 코드만으로 실제 리전을 확정할 수 없으므로 계약·DPA와 콘솔 설정을 확인해야 한다.

### 7.4 세무대리·회계법인 자격

[공인회계사법 제2조](https://www.law.go.kr/lsLinkProc.do?chrClsCd=010202&joLnkStr=%EC%A0%9C2%EC%A1%B0+%EB%98%90%EB%8A%94+%EC%A0%9C40%EC%A1%B0%EC%9D%983&joNo=000200000%5E004000000%5E004002000%5E004003000&lsId=001592&lsNm=%EA%B3%B5%EC%9D%B8%ED%9A%8C%EA%B3%84%EC%82%AC%EB%B2%95&mode=2)은 공인회계사의 직무에 세무대리를 포함한다. 다만 실제 세무대리는 필요한 등록과 수임을 갖춘 공인회계사·회계법인 또는 다른 법정 자격자가 수행해야 한다.

[대법원 무자격 세무대리 판례](https://www.law.go.kr/LSW/precInfoP.do?precSeq=226329)는 무자격자가 자격자의 지휘·감독 없이 납세자를 대리하거나 사실상 신고를 주도하는 경우를 문제 삼는다. 소프트웨어 공급자가 계산·판단·제출을 대신 운영하거나 고객의 홈택스 계정을 원격으로 다루면 단순 도구 공급을 넘어갈 위험이 있다.

따라서 계약과 운영에서 다음을 지킨다.

- 판매대상은 자격·등록·수임·EDI 업무대행 지정이 확인된 조직과 사용자
- 공급자는 세액 판단·신고서 확정·최종 제출을 하지 않음
- 원격지원자는 고객 계정에 대신 로그인하지 않음
- 고객의 권한 있는 담당자가 결과 검토와 제출 책임을 부담하되, 공급자의 고의·중과실이나 보안결함까지 전부 고객에게 전가하지 않음

## 8. 저작권·오픈소스·상표

### 8.1 PyMuPDF — 독립적인 출시 차단 항목

`requirements.txt:2`는 `PyMuPDF>=1.23`을 요구하고, `build.py:133-137`, `build.py:247`, `build.py:268`은 PyMuPDF와 `_mupdf.pyd`가 상용 번들에 실제 포함되었는지를 의도적으로 검증한다. `src/utils/pdf_reader.py:4`에서 실제 사용한다.

[PyMuPDF 공식 문서](https://pymupdf.readthedocs.io/en/latest/faq/index.html)는 PyMuPDF/MuPDF가 GNU AGPL 또는 Artifex 상용 라이선스로 제공됨을 안내한다. 폐쇄형 설치 프로그램을 AGPL 방식으로 배포하려면 전체 결합물에 미치는 의무를 전문 변호사가 검토하고 실제로 이행해야 한다. 소스 공개를 원하지 않는 상용 제품이라면 현실적인 선택지는 다음 둘이다.

1. Artifex 상용 라이선스를 계약하고 해당 버전·제품·배포 범위를 문서화
2. 기능을 허용적인 라이선스의 라이브러리로 교체하고 빌드에서 PyMuPDF가 완전히 제거됐음을 검증

정적 호출 검색에서는 `src/utils/pdf_reader.py`를 다른 제품 코드가 호출하는 경로를 찾지 못했고, 실제 NHIS 원시자료 처리는 `src/utils/raw_data_reader.py:49-80`의 `pdfplumber` 경로를 사용한다. 따라서 가장 빠른 해결책은 PyMuPDF가 정말 불필요한지 회귀 테스트한 뒤 `requirements.txt`, `build.py`의 강제 수집·검증, 미사용 모듈에서 완전히 제거하는 것이다. 단순히 import만 안 하거나 설치본에 남겨 두는 것은 배포 문제를 해결하지 못한다.

이 문제는 포털 약관과 무관하며 판매 전에 반드시 해결해야 한다.

### 8.2 PySide6·기타 의존성

[Qt for Python 공식 문서](https://doc.qt.io/qtforpython-6)는 PySide6를 LGPLv3/GPLv3 또는 Qt 상용 라이선스로 제공한다고 명시한다. LGPL로 배포한다면 사용 모듈별 고지, 라이선스 사본, 수정 여부, 역공학을 금지하지 않는 계약, 교체·재링크 가능성 등 배포 방식에 맞는 의무를 정확히 설계해야 한다. 단순히 “PySide는 무료”라고 처리하면 안 된다.

[Qt의 공식 LGPL 의무 안내](https://www.qt.io/development/open-source-lgpl-obligations)에 따라 정확한 Qt/PySide/Shiboken 버전과 소스·수정사항을 제공하거나 유효한 제공 방식을 마련하고, 사용자가 LGPL 라이브러리를 수정·교체·재링크하여 실행할 수 있게 해야 한다. EULA에는 그 목적의 역공학·디버깅을 금지하지 않는 예외가 필요하다. 현재 onedir 구조는 DLL 교체 가능성에 유리하지만, 고지·소스·교체 절차·실제 테스트가 없는 상태이므로 그것만으로 준수가 끝나지 않는다.

현재 `requirements.txt`는 최소버전만 지정하고 상한·해시가 없어 나중 빌드에서 코드와 라이선스가 달라질 수 있다. 전체 의존성 잠금, SBOM, `THIRD_PARTY_NOTICES`, 원문 라이선스 동봉, 빌드별 아카이브가 필요하다. PyInstaller의 예외가 PyMuPDF나 Qt의 라이선스 의무까지 없애 주는 것은 아니다.

[PyInstaller 공식 라이선스 예외](https://pyinstaller.org/en/stable/license.html)는 생성된 상용·폐쇄형 실행파일 배포를 허용하므로 stock PyInstaller 자체는 현재 출시 차단 요인이 아니다. 다만 그 예외는 번들 안의 각 라이브러리 라이선스를 변경하지 않는다.

### 8.3 포털 UI·상표

WEHAGO, 홈택스, 각 공단의 로고·화면·명칭을 제품명이나 광고에서 공식 제휴처럼 사용하면 안 된다. 설명적 호환성 표시는 최소한으로 하고 “비공식·비제휴 독립 소프트웨어”를 명확히 표시한다. 사이트 UI 자산·스크립트를 제품에 복제·포함하지 않는다.

2021도1533 판결은 데이터베이스의 전부·상당 부분 또는 작은 부분의 반복적·체계적 복제가 데이터베이스제작자의 통상적 이용과 충돌하는 경우 별도의 권리침해가 될 수 있음을 설명한다. 현재처럼 적법하게 수임한 개별 사업장의 신고·보험 문서를 업무 목적으로 처리하는 것은 경쟁 서비스의 데이터베이스를 대량 복제해 재판매하는 사안과 다르므로 위험이 상대적으로 낮다. 그래도 공급자 중앙서버에 여러 법인의 자료를 모아 데이터상품·벤치마크로 재사용하는 기능은 별도 동의만으로 섣불리 추가하지 말고 저작권·개인정보를 다시 검토해야 한다.

## 11. 판매 계약과 운영 문서

시트 구독 상품에는 최소한 다음 문서가 필요하다.

1. **주문서/구독조건**: 법인명, 좌석 수, 허용 기기, 기간, 자동갱신, 가격·세금, 해지·환불
2. **B2B EULA**: 독립 소프트웨어임을 밝히고 재배포·계정공유·우회 사용 금지, 업데이트·지원·종료 조건 규정
3. **개인정보처리방침**: 프로그램 계정, 장치·접속정보, 지원 데이터, Supabase 등 수탁자·국외 이전, 보유·파기
4. **DPA/보안 부속서**: 고객 자료에 접근하는 경우의 목적·범위·재위탁·사고통지·반환·파기
5. **SLA/지원정책**: 포털 개편·점검·약관 변경으로 인한 중단, 긴급 패치, 수동 우회절차
6. **오픈소스 고지**: 사용 버전, 저작권, 라이선스 원문, 소스·교체수단 제공이 필요한 항목
7. **사용자 운영지침**: 권한·위임 확인, 사람 로그인, 승인·접수증 보관, 데이터 파기, 퇴사자 세션 폐기

순수한 법인 간 거래인지, 개인 세무사·소규모 사업자 또는 소비자도 온라인으로 가입할 수 있는지에 따라 전자상거래·통신판매 규율의 적용 범위가 달라질 수 있다. 판매 화면에는 사업자 신원, 총가격·부가세, 최소기간, 자동갱신, 좌석 증감 기준, 해지·환불, 포털 차단 시 기능중단·대체수단을 결제 전에 명확히 표시하고 동의 버전을 보존해야 한다. 정기결제 금액 인상이나 무료에서 유료 전환은 별도 고지·동의 절차를 법무 검토한다. 좌석 통제를 위해 장치지문을 과도하게 수집해서도 안 된다. [2026-07-17 현재 전자상거래법](https://www.law.go.kr/LSW/lsInfoP.do?ancYnChk=0&chrClsCd=010202&efYd=20260120&lsiSeq=282793&urlMode=lsInfoP)의 적용 여부는 실제 고객·판매채널별로 확정하고, 일부 개정 시행일인 2026-07-21 이후 출시본에서는 법령을 다시 확인한다.

면책조항만으로 설계 결함이나 고의·중과실을 고객에게 전가할 수 있다고 가정하면 안 된다. 특히 신고 오류·기한 도과·개인정보 유출 가능성을 알고도 충분한 안전장치를 만들지 않은 경우에는 광범위한 면책이 실무상 방어력이 약하다.

## 12. 기관에 보낼 서면 질의의 핵심 내용

구두 고객센터 답변이 아니라 담당 부서의 이메일·공문·계약으로 남기는 것이 좋다. 질의에는 유리한 사실만 쓰지 말고 실제 제품 범위를 정확히 설명해야 한다.

### NHIS·NPS

- 정식 업무대행 지정과 수임을 받은 회계법인 직원이 직접 인증·로그인함
- 고객 PC의 로컬 프로그램이 위탁사업장을 선택하고 결정내역 PDF/XLS를 저속으로 다운로드함
- 인증정보를 저장하지 않고 CAPTCHA·IP 우회가 없으며 공급자 서버로 업무자료를 보내지 않음
- 여러 회계법인에 유상 배포하고, 일부 값은 고객의 WEHAGO 급여업무에 입력함
- 이 시나리오가 “사전 허락” 대상인지, 허락 신청 주체가 공급자인지 각 회계법인인지, 허용 수임처·동시성·속도·자료 재사용 범위가 무엇인지 질의

### WEHAGO

- 고객 직원이 직접 로그인한 Chrome을 로컬 설치 프로그램이 제어함
- 수임처 조회, 급여 Excel 다운로드·업로드, PDF 저장, 신고 마감, 전자신고 파일 생성 기능을 유상 공급함
- 직접 서버 API 호출·인증정보 저장·CAPTCHA 우회는 하지 않음
- 허용 여부, 공식 가져오기/API·파트너 경로, 요청 간격, 계정·좌석·고객지원 조건을 질의
- 현재 사용하는 내부 React/RealGrid 조작은 허락 전 제거하는 것이 바람직함

### 국세청

- 정식 세무대리인이 변환신고 `.01` 파일을 선택·검증하는 로컬 보조도구
- 인증은 사용자가 하고 최종 제출은 홈택스 원래 화면에서 직접 수행하도록 설계
- 상용 배포가 이용약관 제15조의 영리행위 제한에 해당하는지, 허용되는 자동화 범위와 공식 연계 경로를 질의

### COMWEL

- 보험사무대행기관 지정·위임을 받은 직원의 로컬 브라우저 자동화
- 자동 다운로드 항목, 수임처 수, 실행 빈도, 데이터의 WEHAGO 재입력 여부를 밝힘
- 약관상 자동화·영리 이용·자료 가공·제공의 허용 범위와 사전승인 절차를 질의

회신에는 제품명·버전·기능 명세·시퀀스·요청량·데이터 흐름을 첨부하고, 승인 범위를 기능 플래그와 계약 고객 범위에 그대로 반영해야 한다.

## 13. 현실적인 출시안

| 단계 | 판매 범위 | 판정 |
|---|---|---|
| 0 | 수동 반출 파일의 로컬 병합·검증·변환만 | P0의 OSS·보안·계약 해결 후 가장 먼저 가능 |
| 1 | WEHAGO 공개 UI 기반 보조, 파괴적 변경은 개별 승인 | 더존 확인·법률검토·stealth 제거 후 제한 파일럿 |
| 2 | 홈택스 파일 선택·검증, 최종 제출은 사람 | 국세청 확인과 안전 UI 후 제한 파일럿 |
| 3 | COMWEL 자동수집 | 최신 약관·권한·필요 시 서면 허락 후 |
| 4 | NHIS·NPS 자동수집 | 각 공단의 명시적 사전 서면 허락 전에는 출시 금지 |

가장 중요한 사업적 메시지는 “전 기능을 한꺼번에 팔 수 있느냐”가 아니라 “약관과 데이터 위험이 낮은 기능부터 독립 SKU로 수익화할 수 있느냐”다. 현재로서는 **수동 반출 + 로컬 처리 + 사람 최종 제출**이 가장 방어 가능하고, 공단 자동화는 승인 기반 옵션으로 분리하는 것이 객관적으로 타당하다.

## 14. 최종 Go/No-Go 게이트

다음 질문에 모두 “예”라고 답할 수 있을 때만 다수 회계법인 대상 유료 출시를 승인한다.

- 고위험 포털별로 최신 약관과 고객별 개별계약을 보존했는가?
- 명시적 사전허락이 필요한 NHIS·NPS 기능은 승인받았거나 제품에서 실제 제거됐는가?
- stealth·webdriver 은닉·내부 세션 함수가 배포본에 없음을 빌드 검사로 증명하는가?
- 불필요한 주민등록번호 전체표출이 없고, 파일·DB·토큰이 암호화되며 자동 파기되는가?
- 홈택스 최종 제출과 파괴적 WEHAGO 변경을 수임처별 권한자가 명시적으로 승인하는가?
- PyMuPDF와 Qt 등 모든 배포 라이선스 의무를 해결했는가?
- 고객 계약, 개인정보 문서, DPA, 지원·사고대응, 코드서명, 좌석 통제가 완성됐는가?
- 외부 변호사가 실제 코드·데이터흐름을 전제로 서면 검토했는가?
- 제한 파일럿에서 계정 제재, 중복신고, 데이터 손실, 치명적 오류가 없었는가?

하나라도 아니면 전체 상용 출시는 계속 보류한다.
