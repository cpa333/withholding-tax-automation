## 0. 한 줄 결론

| 구분 | 판정 |
|------|------|
| **전 기능(국민연금 포함) 일괄 판매** | **No-Go에 가까움** — 약관상 명시 금지 모듈을 유상 상품에 넣으면 “약관 위반 도구 판매” 프레임 |
| **조건부 단계 판매 (권고)** | **Conditional Go** — WEHAGO 중심 → 건보/고용 → 홈택스(자기 제출 보조) 순. **국민연금은 서면 허락 전 제외** |

**핵심 메시지:**  
내부 자가사용 리스크와 **제3자 유상 판매** 리스크는 질적으로 다릅니다. 자가사용은 “이용자 약관 위반” 중심이지만, 판매는 약관 위반 유도·표시광고·고객 계정 제재 손해·정책 단속 프레임이 추가됩니다.

- Windows 설치형 `.exe`, Playwright **CDP**로 **실제 Chrome** 제어
- **사람 로그인**(human-in-the-loop): 공동인증서/간편인증은 프로그램이 대신하지 않음
- `human_delay` 등 저부하 지향 설계
- **stealth**: `navigator.webdriver` 숨김 및 playwright-stealth 일부 적용 (`src/utils/stealth.py`)
- 라이선스: **Supabase Auth** (이메일/비번) — 신고·EDI 데이터 중앙 수집 없음 **전제**
- 데이터: 고객 PC 로컬 저장 중심

### 2.2 국민연금 EDI (`edi.nps.or.kr`) — 최대 리스크

**원문 확인 (2024-08-01 시행본, 사이트 약관 모달·전문)**

> **제10조 (서비스의 이용제한 등)** 제10호  
> 서비스 운영시간에 비정상적인 행위(**매크로, 크롤링, 스크래핑**) 등으로 정상적인 서비스 운영에 지장을 주는 경우  
> → 사전통지 없이 이용 중지·관계당국 고발 등 가능

> **제12조 (이용자의 의무)** 제2항 제12호  
> **사전 허락 없이** 자동화된 수단(매크로, 스크래핑 등)을 이용하여 당 사이트가 제공한 서비스에 **로그인을 시도 또는 로그인**하는 행위  
> (**인증정보를 이용하여 로그인 후 개인정보를 수집하는 행위를 포함**)

> **제12조** 제2항 제13호  
> IP를 지속적으로 변경하며 접속 또는 **캡챠(CAPTCHA)를 외부 솔루션 등을 통해 우회하거나 무력화**하는 행위

> **제12조** 제4항  
> 서비스를 이용하여 얻은 정보를 당 사이트의 **사전승낙 없이** 복사·복제·변경·번역·출판·방송 기타 방법으로 사용하거나 **타인에게 제공**할 수 없음

**판단**

| 쟁점 | 결론 |
|------|------|
| 사람 로그인 후 자동 다운로드 | 문언상 **제12호 괄호(“로그인 후 개인정보 수집”)에 걸릴 수 있음**. “해당 없음” 단정 불가 |
| 사전 허락 | 조문상 **가장 확실한 해소 조건** |
| stealth / 봇 탐지 회피 | 제13호·보호조치 우회 서사와 맞닿음 → **상용에서 제거 권고** |
| 상용 판매 | 구매자의 약관 위반을 **설계적으로 전제**한 상품 → 판매자 평판·계약 리스크 최대 |

**대응:** 메인 SKU에서 **국민연금 모듈 제외** 또는 공단 **서면 허락 후 옵션 활성화**. 내부 `국민연금EDI.md`의 “사전 허락 절차” 권고는 상용 판매에서 더욱 필수.

### 2.3 건강보험 EDI (`edi.nhis.or.kr`)

**내부 검토 (`건강보험EDI.md`, 2015-09-01 약관 기준) 요지**

- 매크로·스크래핑 **명시 금지 조항 없음**
- 남는 선: 정상 운영 지장 금지, 얻은 정보의 외부 복제·배포·제3자 제공 금지
- 1인 세무사 업무대행 지정(2026-03-17~) 등 **정당한 대행 자격** 전제

**본 조사 보완**

- EDI 로그인 동의 약관 전문은 자동 수집이 어려워 **출시 직전 화면 캡처 보관** 필요
- 동일 기관(건보공단) 다른 서비스 약관에 “사전 허락 없는 매크로·봇·스크래퍼” 금지 템플릿이 존재 → **EDI도 향후 개정 가능**
- 국민연금 2024 개정처럼 따라올 수 있으므로 **분기별 약관 모니터**

**상용 판매:** Conditional Go (업무대행 지정 고객만, 로컬 저장, 저부하).

### 2.6 WEHAGO (더존비즈온)

| 조문 | 내용 | 판매 함의 |
|------|------|-----------|

| **제8조①7호** | 서비스를 목적 외 상업적으로 **복사·복제·재판매·재임대·증여** 금지 | 금지 주체는 **회원**. WEHAGO 서비스 자체의 재판매를 겨냥. **제3자 로컬 RPA 판매를 직접 금지하진 않음** |

| **제10조④** | 알게 된 정보를 **계약 이행 목적**으로만 | 수임 업무 목적이면 정합; 판매자 중앙 DB 축적 시 위험 |

| **자동화/매크로 금지** | 공개 약관에서 **미발견** | “허용 확인”이 아니라 “명시 금지 없음” |

**기존 WEHAGO 판매 보고서(docx)와의 정합**

- “공개 UI 기반 로컬 RPA 포괄 금지 문구 미확인” → **재확인 일치**
- 유상 판매 자체를 약관이 곧바로 불법으로 만드는 것은 아님
- 다만 **공식 제휴 아님**, 약관 개정·개별 중단 통지·서버 부하 리스크는 잔존

**제8조①7호 vs 제3자 RPA 판매**

- 직접 적용 대상: 대개 **회원(회계법인)**  
- 판매자: 회원이 아닐 수 있어 동 조항 **직접 위반 당사자가 아닐 수 있음**  
- 간접: 고객 제재 → 클레임, “WEHAGO 연동 상품” 마케팅 공격, 향후 이용정책 금지

### 3.1 대법원 2021도1533 (야놀자 크롤링)

### 3.2 자가사용 vs 상용 판매

| | 자가사용 | 시트당 구독 판매 |
|--|----------|------------------|
| 약관 당사자 | 본인 | 구매자 (판매자는 보통 비회원) |

## 4. 개인정보·세무 책임 (요약)

개인정보 전담 서브에이전트는 세션 한도 오류로 미완. 아래는 법령 일반 원칙 + 제품 구조 기반 정리.

### 4.1 판매자가 개인정보 처리자/수탁자가 되는 경우

| 조건 | 지위 가능성 | 위험 |
|------|-------------|------|
| 라이선스 이메일·시트 수만 Supabase | 처리자(최소) | 처리방침·보관·(국외)이전 고지 필요, **관리 가능** |
| 신고파일·EDI PDF·주민번호·급여 중앙 수집 | 처리자/수탁 **고위험** | 위탁계약·안전조치·유출 시 본인 책임 |
| 원격지원 중 화면의 민감정보 열람 | 사실상 처리 관여 | 옵트인·마스킹·기록 |
| 인증서 비번·세션 토큰 수집 | **치명** | 형사·개보 이슈 — 현재 설계와 충돌, 유지 금지 |

**안전한 판매 형태:** 도구 제공자. 수임처 개인정보의 **처리자는 회계법인**.  
**위험한 형태:** “클라우드 OCR·자동 진단 업로드·중앙 원천 DB”.

### 4.2 고객(회계법인) 측

- 수임·업무대행 범위 내 처리가 원칙 (세무대리·4대보험 대행 지정)
- 홈택스 제55조: 수임 세무정보 이용 **동의서 보관** 등 기존 의무 유지
- 자동화로 “수집 양”이 늘면 목적 범위·보관 기간 재점검 권고

### 4.3 세무·신고 책임

- 세액·신고 적정성·가산세: **권한 있는 담당자(고객)**  
- 계약·UI: 자동 결과는 초안, **최종 확인·제출은 사람**  
- 판매자: 고의·중과실 제외한 간접손해·가산세 면책 설계 (변호사 검토)

## 8. “합법적 판매” vs “리스크 관리형 판매”

| | 정의 | 현 제품 |
|--|------|---------|
| **합법적 판매에 가까움** | 공식 API·제휴·사전 허락, stealth 없음, 약관 허용 범위만 상품화 | **미도달** |

**객관적 분류:** 현재 코드 베이스 전체를 그대로 팔면 **리스크 관리형에도 미달(국연·stealth)**. Phase 분리·stealth off 후 **리스크 관리형 Conditional Go**.

## 12. 최종 의견

1. **합법적 판매를 “전 기능 보장”으로 단정할 수 없다.**  
2. **가장 명확한 약관 충돌은 국민연금 EDI 제12조 제2항 제12호**이다. 상용 패키지에 넣으면 리스크 등급이 Critical이다.  
3. **홈택스 파일변환 업로드**는 행정규칙상 **허용된 신고 방식**과 맞닿아 수집형 플랫폼보다 유리하나, 정책·차단·과부하 리스크는 별개이다.  
4. **WEHAGO 로컬 UI 보조**는 공개 약관상 명시 금지가 없어 Phase 1 후보로 가장 현실적이다. 비제휴·저부하·검증 UI가 전제이다.  
5. **stealth는 상용과 양립하기 어렵다.**
