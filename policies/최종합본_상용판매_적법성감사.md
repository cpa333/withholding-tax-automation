## 1. 4개 모델 종합 결론 (한 문단)

**네 모델은 표현("NO-GO" vs "조건부 가능")이 달라 보이지만 실질은 하나로 수렴한다: ⓐ 현재 코드베이스 전체를 그대로, 지금, 다수 회계법인에 유상 판매하는 것은 네 모델 모두 비권장(사실상 NO-GO)이며, ⓑ 기능을 축소·수정하고 단계적으로 파는 "리스크 관리형 조건부 판매"는 네 모델 모두 현실적으로 가능하다고 본다.** 이 제품은 "본질적으로 불법인 제품"이 아니다 — 대상 사이트 로그인을 사람이 직접 수행하고(human-in-the-loop), 공동인증서 비밀번호를 취급하지 않으며, 판매자 서버로 업무데이터를 전송하지 않고(텔레메트리 0건, 코드 검증됨), 홈택스·근로복지공단은 공식 경로/read-only에 가깝고, 세무사법 리스크도 낮다. 그러나 현재 빌드에는 판매자의 2차 책임을 실제로 활성화할 수 있는 요소가 동시에 존재한다 — **①전역 탐지회피(스텔스) 코드, ②자동화를 명시적으로 금지한 EDI 모듈(국민연금 확정, 건강보험은 2025-10-23 개정으로 재분류), ③홈택스 무인 최종제출, ④개인정보·계약(EULA·처리방침·시트 라이선스) 공백, ⑤PyMuPDF(AGPL) 상용 동봉, ⑥국민연금 주민번호 전체표출 과다수집.** 가장 보수적인 견해(Codex)는 이를 즉시 출시 차단 사유로, 가장 현실적인 견해(Claude/Grok/GLM)는 판매 전 필수 수정 후 단계 출시로 본다. **통합 권고는: 현재 전 기능 즉시 판매는 보류(NO-GO), 위하고 중심 최소 구성부터 스텔스 제거·무인제출 게이트·개인정보 서류·라이선스 의무 이행 후 단계적 조건부 출시.**

### 모델별 최상위 판정 비교표


| 모델         | 최상위 판정(전 기능 즉시 판매)                  | 단계적/수정 후 판정                                           | 이 모델의 결정적 강조점                                                                                              |
| ---------- | ----------------------------------- | ----------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| **Claude** | 현재 코드 그대로 판매 **비권장** (조건부 판매 가능 제품) | **조건부 가능** — 4가지 수정(스텔스/EDI/무인제출/개인정보) 이행 시           | 스텔스+명시금지 사이트+상용판매 조합이 **대법 2024도14169**(무력화 도구 판매자 유죄+추징)로 가는 유일 통로. 건강보험 EDI 2025-10-23 개정으로 재분류          |
| **Codex**  | **NO-GO** (출시 보류)                   | **조건부 가능** — 구조 변경 시(로컬 문서처리 SKU부터)                   | 가장 보수적. **PyMuPDF(AGPL)**·주민번호 전체표출·공단자료 WEHAGO 재업로드·시트라이선스/EULA/코드서명 미비를 독립 출시차단 사유로 추가                   |
| **Grok**   | 전 기능(국민연금 포함) 일괄 판매 **No-Go에 가까움**  | **Conditional Go** — 단계 출시(WEHAGO→건보/고용→홈택스), 국민연금 제외 | 자가사용 vs 판매의 질적 차이를 리스크매트릭스(R1~R11)로 체계화. "합법 보장"은 불가, "리스크 관리형"이 도달 목표                                     |

**합의점(4/4 모델 동일):** ① 국민연금 EDI가 최대 단일 리스크(약관 제12조②12호 명시 금지) ② 스텔스는 상용과 양립 불가, 제거 필요 ③ 단계적·조건부 출시가 유일한 현실 경로 ④ 홈택스 파일변환신고는 국세청고시상 공식 경로여서 수집형 스크래핑보다 유리 ⑤ WEHAGO 공개 약관에 RPA 직접 금지 조항은 없음 ⑥ "100% 합법 보장" 마케팅 불가 ⑦ 출시 전 변호사 서면의견 + 기관 서면질의 필수.

### 4.3 [충돌→REFUTED] "홈택스 최종 제출은 사람이 한다"

원본 전제(및 docx의 "최종 제출 전 사람 확인")와 달리, 코드는 `dry_run=False` 시 제출 버튼 + 확인모달 2회 + 접수증까지 전자동 클릭한다(C5, `_upload.py:251-297`). 4개 모델 모두 이 점을 지적했고 코드로 REFUTED다. 기본값이 dry-run ON이나, **전역 체크박스 해제만으로 건별 확인 없이 되돌릴 수 없는 신고가 나간다.** → 수임처별 세액 요약·사람 명시 확인 게이트로 강제하고 확인모달 자동클릭을 삭제해야 한다(P0-3).

### 4.4 [충돌→부분 REFUTED] "보안장치 미우회 / 우회 기능 미탑재"

원본 전제와 달리 스텔스는 실재한다(C1·C3·C4). 다만 Claude의 적대적 검증 nuance도 코드로 확인된다(C2): **지문 스푸핑·CAPTCHA 해제·IP 순환은 없고**(의도적 OFF), webdriver 은닉과 자동화 아티팩트 스푸핑만 켠 절제형이다. NHIS 플래그의 명시 동기는 "무한 리로드(로그인 루프) 방지=기능 정상화". → "우회 기능 미탑재"는 틀렸고(존재), "지문 스푸핑까지 하는 공격형은 아님"은 맞다. **검증자 경고**: "무한 리로드 차단이 법적으로 접근통제조치라면, 그 무력화는 오히려 정보통신망법 제48조 쪽으로 형사 리스크를 높인다." 형사 등급을 낮춘 전제가 미검증 법적 프레이밍 위에 있으므로, 상용 빌드에서 제거·목적 재문서화가 안전하다.

### 4.5 [Codex NO-GO vs 나머지 조건부] — 실제로 얼마나 다른가

표면적으로 Codex만 "NO-GO"이고 나머지는 "Conditional Go"지만, 문언을 보면 **네 모델의 실질 결론은 동일**하다:

- Codex: "현재 코드 **전체를 그대로** 유상 판매 = NO-GO" + "구조를 바꾸면 조건부 판매 현실적으로 가능".
- Claude: "현재 코드 그대로 판매 비권장" + "필수 수정 이행 시 조건부 판매 가능".
- Grok/GLM: "무조건 전 기능 = No-Go" + "단계적 = Conditional Go".

→ **차이는 판정이 아니라 강조점과 임계값**이다. Codex가 다른 셋보다 더 보수적인 지점은 (a) **PyMuPDF(AGPL)를 독립 출시차단 사유로 격상**(§4.6), (b) 주민번호 전체표출을 최소수집 위반으로 명시, (c) 시트 라이선스·EULA·코드서명·Defender 예외를 "판매 준비 미완"으로 강하게 못박은 것. 이 셋은 다른 모델이 "권고"로 낮게 본 것을 Codex가 "P0"로 올린 것이며, **검증 결과 셋 다 실재하므로 Codex의 격상이 타당**하다. 통합 판정은 Codex의 P0 목록을 채택하되(가장 보수), 출시 경로는 Grok/GLM의 단계론(가장 현실)을 채택한다.

### 4.6 [Codex 독자→CONFIRMED] PyMuPDF(AGPL) 라이선스

Codex만 제기. 검증 결과 CONFIRMED(C7): PyMuPDF는 AGPL 또는 Artifex 상용 이원 라이선스인데, 폐쇄형 유료 설치본에 `build.py`가 강제로 동봉하고 `_mupdf.pyd` 존재까지 검증한다. **동시에 제거가 쉽다** — `pdf_reader.py`(fitz)를 제품 코드가 import하지 않고 실제 PDF 처리는 pdfplumber(BSD 계열)로 이뤄진다. → **가장 빠른 해결은 회귀 테스트 후 requirements/build.py에서 PyMuPDF 완전 제거.** 상용 라이선스 계약도 대안이나 불필요할 가능성이 높다. Qt/PySide6(LGPLv3)의 배포 의무(THIRD_PARTY_NOTICES·소스/재링크 제공)도 Codex가 지적했으며 별도 이행 필요.

### 4.7 [GLM 독자→법리 유효, 판례 실무확인 필요] 공공누리(KOGL)·부정경쟁 차목

GLM만 제기. 레포 무관한 외부 법리이므로 코드 검증 대상은 아니나 결론은 유효하다: ⓐ **KOGL은 기대 불가** — 홈택스/공단 EDI의 신고정보·가입자 자료는 '저작물'이 아니라 개인정보·세무정보이므로 "KOGL Type 1로 출처표시하면 상업 자동화 가능"이라는 논리는 성립하지 않는다(오히려 개인정보보호법·비밀유지의무 규율). ⓑ **부정경쟁방지법 제2조1호 차목**('타인의 기술적 성과 무단이용')은 "부정한 방법으로 취득·사용" 요건을 도구 판매가 충족하기 어려워 위험이 낮다(대법 2019도14725는 데이터 무단수집·DB화 사안). **단 GLM 자신이 판례번호 실무 대조를 권고**했으므로 변호사 확인 항목으로 남긴다(신뢰도 MEDIUM).

### 4.8 [Claude 독자→판례번호 실무확인 필요] 대법 2024도14169

Claude만 인용한 "운영자가 금지·차단하는 기능을 무력화하는 프로그램의 판매자를 구매자와 업무방해 공동정범으로 보아 판매대금 전액 추징"이라는 판례. **스텔스+명시금지 사이트+상용판매 조합의 형사 리스크를 가장 날카롭게 포착**한 근거이나, 레포로 검증 불가하고 다른 3개 모델은 인용하지 않았다. → 이론적 종착점으로서 경고 가치는 크되(스텔스 제거의 사활적 근거), **판례번호·판시 범위는 변호사 원문 대조 필수**(신뢰도 MEDIUM). ### 8.2 기관 서면질의 항목(회신은 방어 근거로 영구 보관)

유리한 사실만 쓰지 말고 실제 제품 범위(사람 직접 로그인, 로컬 실행, 저속, 자격증명 미저장, 판매자 서버 미전송, 다수 법인 유상 배포, 일부 값 WEHAGO 재입력)를 정확히 기술한다.

- **국민연금공단·건강보험공단(EDI 약관 소관)**: 정당한 업무대행 지정·수임을 받은 회계법인 직원의 수동 로그인 후 자기 수임처 자료 화면 자동화 다운로드가 ⓐ (NPS 제12조②12호 / NHIS 개정 제17조) 사전허락 대상인지, ⓑ 허락 신청 주체가 판매자인지 각 회계법인인지, ⓒ 허용 수임처·동시성·속도·자료 재사용·전체표출 범위.
- **근로복지공단(1833-6000)**: 동일 질의 + QR 자동발급 공식채널로 목적 달성 가능 여부 + 토탈서비스 약관 자동화 조항.
- **더존비즈온(WEHAGO)**: 고객 직원 직접 로그인·로컬 설치형 상용 RPA(공개 UI만)의 허용 여부·속도·계정·좌석·지원 조건, 공식 가져오기/API·파트너 경로. 내부 React/RealGrid 조작은 허락 전 제거 전제.
- **국세청**: 정당 세무대리인의 변환신고 `.01` 파일 선택·검증 로컬 보조도구(인증·최종제출은 사람)의 상용 배포가 이용약관 제15조 영리행위 제한에 해당하는지, 허용 자동화 범위·공식 연계 경로.

## 9. 종합 의견 — 가장 보수적 견해 · 현실적 견해 · 통합 권고

**가장 보수적 견해(Codex 축):** 현재 코드 전체는 NHIS·NPS 명시적 자동화 제한, 전역 스텔스, 홈택스 무인 제출, 주민번호 전체표출 보존, PyMuPDF(AGPL) 동봉, 시트 라이선스·EULA·코드서명 미비 때문에 **다수 회계법인 대상 즉시 유료 판매에 부적합(NO-GO)**. 이 중 어느 하나만으로도 출시를 막을 사유가 된다.

**가장 현실적 견해(Claude/Grok/GLM 축):** 이 제품은 "팔 수 없는 제품"이 아니다. 인프라는 깨끗하고(판매자 서버로 업무데이터 미전송·텔레메트리 0건 — 코드 검증), 대상 사이트 로그인을 자동화하지 않으며, 공동인증서 비밀번호를 취급하지 않고, 홈택스·근복은 공식 경로/read-only에 가깝다. 형사 침입·업무방해·악성프로그램의 무죄 판례 계열(2021도1533·2017도16520·2017노309·2005도870)에 상당히 근접해 있어, **판매 전 필수 수정 후 자격자 대상 B2B로 단계적 조건부 판매가 가능**하다.

**통합 권고(가장 보수적이면서 현실적):**

1. **현재 전 기능 즉시 판매 = 보류(NO-GO).** 네 모델이 수렴하는 지점이다.
2. **P0(§6) 14개를 완료**하기 전에는 어떤 유료 출시도 하지 않는다. Codex의 P0 목록(스텔스·EDI·무인제출·PyMuPDF·전체표출·개인정보·코드서명·라이선스)을 채택한다 — 검증 결과 모두 실재한다.
3. **출시 경로는 §7 단계론**을 따른다 — 위하고 중심 Stage 1부터, 두 EDI는 공식 서면 허락 전 기본 비활성화. Grok/GLM의 현실적 단계 접근이다.
4. **가장 큰 실질 리스크는 형사 처벌이 아니라 ①홈택스 무인 제출로 인한 오신고 가산세 손해배상과 ②상류기관(더존·공단·국세청)의 약관 개정·기술 차단으로 인한 제품 지속성**이다. 전자는 사람 최종 확인 게이트로, 후자는 per-feature 원격 비활성화(feature flag)·계약 고지·수동 전환 SLA로 관리한다.
5. **"명시적 금지 사이트(국민연금·건강보험 EDI) + 스텔스 + 상용 판매"의 삼중 조합**이 판매자 형사·민사 책임 고리(2024도14169 계열)를 가장 강하게 만든다. 이 조합을 깨는 것 — 스텔스 제거 + 두 EDI 분리 — 이 "중립적 생산성 도구" 방어의 사활을 가른다.

**요약 한 줄:** 지금 전부 파는 것은 안 된다(NO-GO). 그러나 스텔스를 걷어내고, 두 EDI를 떼고, 홈택스 무인 제출을 막고, PyMuPDF·전체표출·개인정보·라이선스 공백을 메우면, 위하고·홈택스 중심으로 자격자 대상 조건부 판매가 가능한 제품이다 — 단, 두 EDI와 "완전 자동 신고"는 공식 허락 없이 팔지 말 것.
